## 一、整体架构

```
                    ┌─────────────┐
                    │   Nginx     │ :5173
                    │  (前端Vue3) │
                    └──────┬──────┘
                           │ /api 代理
                    ┌──────▼──────┐
                    │  Gin API    │ :8080
                    │  (主进程)   │
                    └──┬───┬───┬──┘
                       │   │   │
          ┌────────────┼───┼───┼────────────┐
          │            │   │   │            │
     ┌────▼───┐  ┌─────▼──▼──▼─────┐  ┌────▼───┐
     │ MySQL  │  │   RabbitMQ      │  │ Redis  │
     │ 8.0    │  │ (异步消息队列)  │  │ 7.x    │
     └────────┘  └────┬─────┬──────┘  └────────┘
                      │     │
               ┌──────▼─────▼──────┐
               │   Worker 进程     │
               │ (异步消费者)      │
               │  - LikeWorker     │
               │  - CommentWorker  │
               │  - SocialWorker   │
               │  - PopularityWorker│
               └───────────────────┘
```

**核心设计思想**：API Server 接收请求后，写操作先发 MQ 消息就返回（快速响应），Worker 进程异步消费消息落库。MQ 不可用时自动降级为直接写 MySQL。

## 三、鉴权系统（JWT 双 Token 模式）

### 3.1 Token 设计
```
Access Token：  JWT (HS256)，15 分钟过期，payload = {account_id, username}
Refresh Token： 32 字节随机 hex 字符串，7 天有效，存在 accounts.refresh_token
```

### 3.2 鉴权流程
```
请求 → 中间件提取 Authorization: Bearer <token>
     → 优先查 Redis: account:<ID>（命中直接通过）
     → Redis 未命中 → MySQL 查 accounts.token 比对
     → 验证通过后回填 Redis（自愈机制）
```

### 3.3 软鉴权 (SoftJWT)
- 不强制要求带 Token
- 但如果带了就必须合法，否则返回 401
- 用于 Feed 流：匿名可浏览，登录态可看点赞/关注状态

### 3.4 Token 失效场景
- 登出：清空 DB 的 token + refresh_token，删除 Redis 缓存
- 改名：重新签发 JWT，旧 Token 失效
- 改密：清空所有 Token，强制所有设备下线

## 四、核心业务逻辑

### 4.1 视频发布（事务 + 发件箱 + 标签提取）

```
1. 上传视频文件 → .run/uploads/videos/{authorId}/{date}/{uuid}.mp4 (最大 200MB)
2. 上传封面图  → .run/uploads/covers/{authorId}/{date}/{uuid}.jpg (最大 10MB)
3. 发布（关键事务）:
   BEGIN TRANSACTION
     INSERT INTO videos (...)
     INSERT INTO outbox_msgs (video_id, event_type="publish", status="pending")
     -- 提取 #话题 标签（正则: #([\p{L}\p{N}_]+)）
     -- 标签不存在则 INSERT，然后 INSERT INTO video_tags
   COMMIT
4. OutboxWorker 异步轮询 outbox_msgs (status=pending)
   → 发布到 timeline MQ → 更新 Redis ZSET → 标记 status=processed
```

### 4.2 点赞/取消点赞（异步优先 + 降级兜底） ⭐重点模式

```
Like(video_id, account_id):
  1. 校验视频存在 + 未重复点赞
  2. 尝试同时发两条 MQ 消息：
     ├── likeMQ.Like()      → like.events (异步写 MySQL likes 表)
     └── popularityMQ.Update(+1) → video.popularity.events (异步更新 Redis 热榜)
  3. 如果两条都成功 → 直接返回（最终一致性）
  4. 如果任一失败 → 降级为事务直写：
     BEGIN TRANSACTION
       INSERT INTO likes (video_id, account_id)
       UPDATE videos SET likes_count = likes_count + 1
       UPDATE videos SET popularity = popularity + 1
     COMMIT
  5. 如果 popularityMQ 失败 → 直接 ZINCRBY Redis 热榜

Unlike 同理，只是 likes_count - 1, popularity - 1（用 GREATEST 防负数）
```

**这个模式的本质**：MQ 是加速路径，MySQL 是保底路径。

### 4.3 评论系统

```
PublishComment(video_id, content, author_id):
  1. 同上 MQ 异步 + 降级模式
  2. 写成功后，扫描 content 中的 @username 提及
     → 对每个被 @ 的用户（非自己）创建 notifications 记录
  3. CommentWorker 消费 commentMQ → 写 MySQL + 更新 popularity + 推送 SSE 通知

DeleteComment: 仅作者可删
```

### 4.4 关注/取关

```
Follow(follower_id, vlogger_id):
  1. 校验双方账号存在
  2. 禁止自己关注自己
  3. 检查是否已关注（防重复）
  4. MQ 异步 + 降级直写（与点赞模式相同）
```

## 五、Feed 流系统（最复杂的部分） ⭐重点

### 5.1 五种 Feed 类型

| 类型 | 路由 | 排序依据 | 分页方式 |
|------|------|----------|----------|
| 最新视频 | `/feed/listLatest` | create_time DESC | 游标 (latest_time) |
| 点赞排行 | `/feed/listLikesCount` | likes_count DESC | 复合游标 (likes_count + id) |
| 热度榜 | `/feed/listByPopularity` | popularity DESC | 快照 + offset |
| 关注流 | `/feed/listByFollowing` | 关注者视频的 create_time | 游标 |
| 标签流 | `/feed/listByTag` | create_time | 无分页 |

### 5.2 三级缓存架构（视频实体获取）

```
GetVideoByIDs(ids[]):
  ┌─ L1: 本地内存缓存 (go-cache, TTL 3s)
  │   命中 → 直接返回
  │   未命中 ↓
  ├─ L2: Redis (video:entity:{id}, TTL 1h)
  │   命中 → 回写 L1 → 返回
  │   未命中 ↓
  └─ L3: MySQL
      命中 → 异步回写 L2 + 回写 L1 → 返回
      每层用 singleflight 防止缓存击穿（同一 ID 并发查询合并为一次）
```

### 5.3 最新视频流：冷热分离 + 游标分页 ⭐核心算法

```
ListLatest(limit, latestBefore, viewerAccountID):

  ┌─ 检查 Redis ZSET feed:global_timeline 是否为空
  │
  ├─【ZSET 为空】→ 全局锁重建（singleflight 防惊群）
  │    1. 从 MySQL 捞最新 1000 条视频
  │    2. ZADD 到 feed:global_timeline (score=create_time.UnixMilli)
  │    3. 递归调用自己（重新走热路径）
  │
  └─【ZSET 不为空】
       对比请求游标 latestBefore 和 ZSET 最老数据的时间 watermark:
       
       ├─ latestBefore > watermark → 【热数据路径】
       │    1. ZRevRangeByScore 从 Redis ZSET 取 ID 列表
       │    2. 三级缓存获取视频实体
       │    3. 如果返回值不足 limit → 拼接冷数据（用 singleflight 防并发）
       │
       └─ latestBefore <= watermark → 【冷数据路径】
            1. 直接查 MySQL（不回写 ZSET，防止冷数据污染热榜）
            2. singleflight 防并发
```

**冷热分离的意义**：Redis ZSET 只存最新数据（约 1000 条），翻到很旧的内容直接查 DB，不污染 Redis。

### 5.4 热度榜：滑动窗口 + 快照分页 ⭐核心算法

```
# 写入侧（Like/Comment Worker）
每次点赞/评论 → ZINCRBY hot:video:1m:{YYYYMMDDHHMM} videoID +1

# 查询侧
ListByPopularity(limit, asOf, offset):
  1. 确定 as_of 分钟（首次请求用当前分钟）
  2. 取最近 60 个分钟窗口的 Key：
     hot:video:1m:{t}, hot:video:1m:{t-1m}, ..., hot:video:1m:{t-59m}
  3. ZUNIONSTORE (SUM 聚合) → hot:video:merge:1m:{asOf}
     设置 TTL 2 分钟（给翻页留时间）
  4. ZRevRange(dest, offset, offset+limit-1) → 取 videoIDs
  5. 查视频实体 → 返回

  # 如果 Redis 不可用 → 降级为 MySQL 游标分页 (popularity, create_time, id)
```

**为什么用快照**：热度实时变化会导致翻页时"第2页出现第1页看过的东西"（跳页/重复）。快照冻结了某个时间点的排名，翻页期间数据不变。

### 5.5 关注流：缓存 + 分布式锁防击穿

```
ListByFollowing(limit, viewerID, before):
  cacheKey = "feed:listByFollowing:limit={n}:accountID={id}:before={t}"
  
  1. 查 Redis 缓存
  2. 命中 → 直接返回
  3. 未命中 → SETNX 抢锁 "lock:{cacheKey}" (500ms TTL)
     抢到锁 → 查 DB → 写缓存 → 释放锁
     没抢到 → 等 5 次 × 20ms 重试读缓存 → 超时则直接查 DB
```

### 5.6 点赞排行：复合游标分页

```
分页游标 = (likes_count_before, id_before)
SQL: WHERE (likes_count < ? OR (likes_count = ? AND id < ?))
     ORDER BY likes_count DESC, id DESC
     LIMIT n

为什么需要复合游标：点赞数相同时，单用 likes_count 会导致
  跳页/重复。加上 id 作为第二排序键保证排序稳定性。
```

## 六、消息队列拓扑（RabbitMQ）

```
Exchange: like.events (topic)
  Queue: like.events, Binding: like.*
    → LikeWorker: 消费后写 MySQL likes 表 + 更新计数

Exchange: comment.events (topic)
  Queue: comment.events, Binding: comment.*
    → CommentWorker: 消费后写 MySQL comments 表 + 更新热度

Exchange: social.events (topic)
  Queue: social.events, Binding: social.*
    → SocialWorker: 消费后写 MySQL socials 表

Exchange: video.popularity.events (topic)
  Queue: video.popularity.events, Binding: video.popularity.*
    → PopularityWorker: 消费后 ZINCRBY Redis 热榜

Exchange: video.timeline.events (topic)
  Queue: video.timeline.update.queue, Binding: video.timeline.*
    → TimelineConsumer: 更新全局时间线 ZSET

死信机制: 每个 Queue 配置 x-dead-letter-exchange = "dlx.events"
  最大重试 3 次，失败消息进入 DLX 防止丢失
```

## 七、实时通知（SSE）

```
通知产生路径:
  Like/Comment/Social Worker 消费 MQ 消息
    → 查找被操作对象（视频作者 / 被提及用户 / 被关注者）
    → INSERT INTO notifications
    → 推送到 SSE Hub（内存中的 map[userID][]chan）

SSE 连接:
  GET /notification/stream?token=xxx
    → 服务端为该用户创建一个 channel
    → 30s 心跳保活
    → 有新通知时写入 channel → 推送到客户端

API:
  POST /notification/list       → 分页查询通知列表
  POST /notification/markRead   → 标记已读（可单条/全部）
  POST /notification/unreadCount → 获取未读数
```

## 九、关键技术决策总结

| 决策 | 说明 |
|------|------|
| 所有接口用 POST | 统一参数传递方式，简化前端调用 |
| Redis 可选依赖 | 连不上自动降级走 MySQL，恢复后自愈回填缓存 |
| RabbitMQ 可选依赖 | 连不上时写操作降级为直接写 MySQL 事务 |
| 软删除不使用 | 所有删除都是硬删除 |
| 没有独立迁移工具 | 依赖 GORM AutoMigrate 自动同步表结构 |
| 双进程部署 | API Server 和 Worker 可独立伸缩 |
| SSE 而非 WebSocket | 单向推送通知，SSE 更简单，兼容性更好 |

## 十一、Python 重构实施路线图

### 11.1 学习理念

重构不是一次性照搬全部功能，而是**分层递进**——每个阶段在前一阶段的基础上增加一个新概念，每完成一个阶段就有一个可运行、可测试的系统。

> 核心原则：**跑通 > 完善，理解 > 照抄**。每个阶段只引入一个新技术或一个新设计模式，避免一口吃成胖子。

### 11.2 阶段划分总览

```
阶段1: 基础 CRUD        → 只涉及 Web框架 + 数据库
阶段2: 鉴权系统          → JWT 双 Token，中间件概念
阶段3: 点赞 + 评论        → 业务事务，外键关联查询
阶段4: Feed 分页          → 三种分页模式（游标/复合游标/快照）
阶段5: 缓存体系          → Redis 多级缓存 + 防击穿 + 冷热分离
阶段6: 热榜系统          → Redis ZSET + 滑动窗口 + 快照聚合
阶段7: 消息队列          → RabbitMQ 异步 + 降级兜底模式
阶段8: 实时通知          → SSE 推送
阶段9: 关注 + 私信        → 巩固前序概念，独立实现
阶段10: 工程化收尾        → Docker、限流、测试、部署
```

### 11.3 各阶段详细说明

---

#### 阶段 1：打地基 — 项目骨架 + 用户 CRUD

**目标**：搭好项目骨架，让一个用户注册/登录能跑通。

**涉及技术**：
- FastAPI 项目结构（router / service / repository 分层）
- SQLAlchemy 模型定义 + Alembic 自动迁移
- bcrypt 密码哈希
- Pydantic 请求/响应序列化

**实现功能**：
| 优先级 | 接口 | 说明 |
|--------|------|------|
| 必须 | `POST /account/register` | 注册，密码 bcrypt 加密入库 |
| 必须 | `POST /account/login` | 登录，先返回一个简单占位 token（阶段2再替换） |
| 建议 | `POST /account/findByID` | 按 ID 查用户 |
| 可选 | `POST /account/findByUsername` | 按用户名查 |

**教学要点**：
- 为什么密码不能明文存储？（彩虹表攻击 → bcrypt 加盐）
- repository 层的作用：把 SQL 和业务逻辑解耦，方便以后换数据库或加缓存
- Pydantic 的 `model_validate` 如何防止客户端传多余字段
- 为什么 Go 项目全部用 POST？Python 重构时是否用 RESTful（GET/POST/PUT/DELETE）？——各有利弊，讨论 RESTful 语义

**验收标准**：用 curl 或 Swagger UI 完成注册→登录全流程。

---

#### 阶段 2：鉴权体系 — JWT 双 Token

**目标**：把阶段1的占位 token 替换为完整的 JWT + Refresh Token 鉴权。

**涉及技术**：
- PyJWT 签发/验证 Access Token (15min)
- refresh_token 随机生成 + 存库 (7天)
- FastAPI Dependency 做鉴权中间件
- 软鉴权（SoftAuth）：不强制带 token，但带了必须合法

**实现功能**：
| 优先级 | 接口 | 说明 |
|--------|------|------|
| 必须 | 改造 `/account/login` | 返回 access_token + refresh_token |
| 必须 | `POST /account/refresh` | 用 refresh_token 换新的 access_token |
| 必须 | `POST /account/logout` | 清空两个 token |
| 建议 | 鉴权中间件 `get_current_user` | FastAPI Depends 注入，保护后续所有接口 |

**教学要点**：
- Access Token 为什么短（15min）？Refresh Token 为什么长（7d）？——减少泄露窗口
- JWT 的三个部分：header.payload.signature 各自的作用
- HS256 vs RS256 的区别和使用场景
- 为什么 logout 要清掉 DB 里的 token？因为 JWT 是无状态的，不存 DB 就无法主动失效
- 软鉴权的实现原理：`try/except` 包裹 token 解析，失败不 abort，只把 user 设为 None

**验收标准**：登录后带 token 访问受保护接口成功；不带 token 访问返回 401；刷新 token 后旧 access_token 仍可用但 refresh_token 只能用一次（Token 轮转）。

#### 阶段 4：Feed 流分页 — 三种分页模式

**目标**：实现 Feed 流的三种分页，这是后端最重要的数据查询技巧。

**涉及技术**：
- 游标分页（cursor-based pagination）
- 复合游标分页（多字段排序的稳定分页）
- offset 分页的优缺点

**实现功能**：
| 优先级 | 接口 | 说明 |
|--------|------|------|
| 必须 | `POST /feed/listLatest` | 游标分页 (按 create_time) |
| 必须 | `POST /feed/listLikesCount` | 复合游标分页 (按 likes_count + id) |
| 建议 | `POST /feed/listByTag` | 按标签筛选 |

**教学要点**：
- **游标分页 vs offset 分页**：游标分页用 `WHERE create_time < ?` 替代 `OFFSET`，避免深分页性能问题。为什么 OFFSET 10000 慢？——数据库仍需扫描并跳过前 10000 行
- **复合游标的必要性**：当排序字段可能重复时（如点赞数相同），只用一个字段会导致 `WHERE likes_count < ?` 跳过不该跳过的行。加入 `id` 作为第二排序键解决问题：
  ```sql
  WHERE (likes_count < ?) OR (likes_count = ? AND id < ?)
  ORDER BY likes_count DESC, id DESC
  ```
- **为什么要用 `?` 占位符而不是 f-string 拼 SQL**：防止 SQL 注入，这是后端开发的安全底线

**验收标准**：前端可以无限下滑加载 Feed 流，不会出现重复或漏数据。

#### 阶段 5：缓存体系 — Redis 多级缓存 + 防击穿

**目标**：引入 Redis 缓存，实现视频详情三级缓存和 Feed 流缓存。

**涉及技术**：
- Redis STRING 缓存序列化（JSON）
- 三级缓存：本地内存 → Redis → MySQL
- 缓存穿透/击穿/雪崩的概念和解决方案
- 分布式锁 SETNX （互斥回源）

**实现功能**：
| 优先级 | 实现内容 | 说明 |
|--------|----------|------|
| 必须 | 视频详情缓存 `video:entity:{id}` | L1 本地缓存 + L2 Redis + L3 MySQL 三级架构 |
| 必须 | Feed 匿名流缓存 | `feed:listLatest:limit={n}:before={t}` TTL 5s |
| 建议 | 关注流缓存 + 分布式锁防击穿 | SETNX 抢锁，未抢到者等待后重试读缓存 |
| 建议 | 鉴权 Token 缓存 `account:{id}` | 登录时写入，鉴权中间件优先读 Redis |

**教学要点**：
- **缓存穿透**：查一个不存在的数据，每次都穿到 DB。解决：布隆过滤器或缓存空值
- **缓存击穿**：热点 key 过期瞬间大量并发回源。解决：分布式锁（SETNX），只让一个请求去查 DB
- **缓存雪崩**：大量 key 同时过期。解决：TTL 加随机值
- **Cache-Aside 模式**：读（先查缓存，miss 查 DB 回填）→ 写（先写 DB，删缓存）— 这是最经典的缓存策略
- 为什么不先删缓存再写 DB？——并发场景下可能出现脏数据（删缓存 → 另一个请求读旧 DB 数据回填缓存 → 写 DB 完成，缓存里是旧数据）。所以正确顺序是**先写 DB 再删缓存**

**验收标准**：用 `redis-cli MONITOR` 观察缓存命中/回源行为，确认热数据走缓存、冷数据回源 DB。

#### 阶段 6：热度排行榜 — Redis ZSET 滑动窗口

**目标**：实现实时热度排行榜，学懂 Redis Sorted Set 的高级用法。

**涉及技术**：
- Redis ZADD / ZINCRBY / ZUNIONSTORE / ZREVRANGE
- 滑动窗口（按分钟分桶）
- 快照分页（保证翻页一致性）

**实现功能**：
| 优先级 | 实现内容 | 说明 |
|--------|----------|------|
| 必须 | 热度写入 | 点赞/评论时 ZINCRBY `hot:video:1m:{minute_key}` |
| 必须 | 热度查询 | 合并 60 个分钟窗口 → 快照 → 翻页 |
| 必须 | `POST /feed/listByPopularity` | 热榜接口 |

**教学要点**：
- **为什么按分钟分桶？** 如果所有热度写同一个 key，高并发下竞争激烈。按分钟分桶把写入分散到不同 key，牺牲一点点查询复杂度换取写入性能
- **ZUNIONSTORE 快照为什么设 TTL 2 分钟？** 快照是临时聚合结果，过期后自动删除。2 分钟足够用户翻几页，过期后下次请求重新生成（拿到更新的排名）
- **快照如何解决翻页抖动？** 不使用快照时：翻第 2 页的过程中，第 1 页的最后一条可能被挤出排名 → 第 2 页出现重复。快照冻结了 `as_of` 时刻的排名，翻页期间数据不变
- ZSET 的底层数据结构是跳表（skiplist），查询 O(log n)，理解它为什么适合排行榜场景

**验收标准**：连续点赞几条视频，观察热榜排名实时变化。翻页不出现重复或缺失。

#### 阶段 7：消息队列 — RabbitMQ 异步 + 降级兜底 ⭐最关键阶段

**目标**：引入 RabbitMQ，把点赞/评论的写操作从"同步事务"升级为"异步优先 + 降级兜底"模式。这是整个项目中最有教学价值的后端设计模式。

**涉及技术**：
- RabbitMQ Topic Exchange + Queue + Binding
- aio-pika 异步消费
- 死信队列（DLX）重试机制
- "异步优先 + 降级兜底" 双路径模式

**实现功能**：
| 优先级 | 实现内容 | 说明 |
|--------|----------|------|
| 必须 | Like/Comment Producer | API 层发布事件到 MQ，成功则立即返回 |
| 必须 | Like/Comment Worker | 独立进程消费 MQ 消息，异步写 MySQL |
| 必须 | 降级兜底逻辑 | MQ 发布失败 → 回退为直接写 MySQL 事务 |
| 建议 | Popularity Producer/Worker | 热度更新也走 MQ |
| 建议 | DLX 死信重试 | 消费失败 3 次后进入死信队列，人工处理 |

**教学要点**：
- **为什么要异步？** API 请求如果同步做所有事（写 MySQL + 更新计数 + 更新热榜），响应时间 = 3 次 I/O 之和。异步后：API 只发 MQ 消息就返回，响应时间 = 1 次 MQ 发布
- **为什么需要降级兜底？** MQ 不是 100% 可靠的（网络抖动、Broker 重启）。如果用户点了赞但数据没落地，这是不可接受的。所以 **MQ 是加速路径，MySQL 直写是保底路径**
- **最终一致性 vs 强一致性**：异步模式下，点赞后立刻刷新 Feed 可能看不到最新点赞数（数据还没被 Worker 消费）。这是可以接受的用户体验优化，代价是短暂的不一致
- **死信队列的作用**：消息消费失败不能直接丢弃——进入 DLX 等待人工排查。常见失败原因：数据库连接断开、外键约束冲突
- **Worker 独立部署的意义**：高峰期 Worker 可以单独扩容，不和 API Server 竞争资源

**验收标准**：
1. 正常情况：点赞 → MQ → Worker 消费 → 数据落库（延迟 < 1 秒）
2. 关闭 RabbitMQ：点赞 → MQ 失败 → 自动降级直写 MySQL → 数据正常落库
3. 关闭 RabbitMQ 后重启：Worker 恢复消费，数据无丢失

#### 阶段 8：实时通知 — SSE 推送

**目标**：实现 Server-Sent Events 实时推送通知。

**涉及技术**：
- SSE 协议（`text/event-stream`）
- asyncio.Queue 做内存消息通道
- sse-starlette 库集成

**实现功能**：
| 优先级 | 实现内容 | 说明 |
|--------|----------|------|
| 必须 | SSE 连接端点 | `GET /notification/stream` |
| 建议 | SSE Hub | 内存 `dict[user_id, list[Queue]]` 管理连接 |
| 建议 | 通知列表/已读/未读 | 3 个辅助查询接口 |
| 建议 | Worker 产生通知 | 点赞/Comment Worker 消费时自动创建通知 + 推送 SSE |

**教学要点**：
- **SSE vs WebSocket**：SSE 是单向（服务器→客户端），WebSocket 是双向。通知场景只需要单向，SSE 更简单且自动重连
- **SSE vs 轮询**：轮询每 N 秒发一次 HTTP 请求，大部分请求是空返回（浪费）。SSE 是长连接，有数据才推送
- **内存 Hub 的局限性**：服务重启 → 所有连接断开。生产环境需要 Redis Pub/Sub 做跨实例消息路由
- **心跳保活**：30 秒发一次 `:keepalive`（SSE 规范中冒号开头的行是注释），防止代理/负载均衡器关闭空闲连接

**验收标准**：用户 A 点赞用户 B 的视频 → 用户 B 的浏览器实时弹出通知。

### 11.4 每个阶段建议的学习节奏

```
1. 读本项目核心文档中对应章节（10 分钟）
2. 读原 Go 项目对应源码，理解逻辑（20 分钟）
3. 关掉原项目，自己用 Python 写（2-4 小时）
4. 遇到卡点先查 Python 官方文档，再参考原 Go 项目（30 分钟）
5. 跑通后用 curl 测试所有接口，写在笔记里（15 分钟）
6. 记录这个阶段学到的 1-2 个核心概念（10 分钟）
```

**关键纪律**：
- **先理解，再动手**。不要一边看 Go 代码一边逐行翻译成 Python，那样你学会的是"翻译"而不是"设计"
- **每阶段只引入一个新概念**。比如阶段 3 不要碰缓存，阶段 4 不要碰 MQ。概念叠加会让你不知道哪个部分出了 bug
- **遇到问题先想"Go 项目为什么这样做"，再想"Python 怎么做最地道"**。Go 的 goroutine + channel 不等于 Python 的 asyncio + Queue，语法翻译 ≠ 设计翻译
