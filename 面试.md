### 面试中"一句话概括项目"的参考答案

> 这是一个仿抖音的短视频Feed流系统，后端采用 Go 的 Gin 框架，核心架构是**双进程部署**——API Server 接收用户请求，Worker 异步消费 RabbitMQ 消息落库。系统最大的亮点是**异步优先+降级兜底**的设计模式：正常情况点赞/评论走 MQ 异步写库实现快速响应，MQ 不可用时自动降级为直接写 MySQL 事务保证数据不丢。Feed 流部分实现了**冷热分离**——最新 1000 条视频缓存到 Redis ZSET，超过范围的请求直接查 DB 避免污染热点数据。热度榜采用**滑动窗口+快照分页**解决翻页抖动问题。缓存体系是**三级架构**——本地内存→Redis→MySQL，每层用 singleflight 防缓存击穿。

### A. 设计决策

#### A.1 系统要解决什么问题？

这是一个**短视频 Feed 流系统**，核心功能是：用户可以刷视频（Feed 流）、点赞、评论、关注、发私信。从后端视角，这本质上是三件事：

1. **内容的写入**：发布视频、点赞、评论、关注 —— 写操作占比高、需要快速响应
2. **内容的阅读**：刷 Feed 流是最核心的读操作 —— QPS 最高、需要极致优化
3. **状态的同步**：点赞数、关注数、通知 —— 允许短暂延迟但不能丢失

#### A.2 为什么选择"双进程"架构？

```
┌──────────────┐        ┌──────────────────┐
│  API Server  │        │     Worker       │
│  (Gin :8080) │        │  (MQ Consumer)   │
│              │        │                  │
│  处理用户请求  │  MQ    │  异步消费消息     │
│  快速返回响应  │──────→│  写MySQL/Redis   │
│              │        │  推送SSE通知      │
└──────┬───────┘        └──────────────────┘
       │
       ├── MySQL (持久化)
       ├── Redis (缓存/热榜/限流)
       └── RabbitMQ (异步消息)
```

**为什么是两个进程而不是一个？**

- **操作系统层面**：API Server 是 I/O 密集型（处理大量 HTTP 连接），Worker 是 CPU/DB 密集型（批量写库、更新热榜）。分离后可以独立配置 CPU 亲和性和内存限制，互不干扰。
- **部署灵活性**：高峰期加大 Worker 实例数来消化消息积压，API Server 不用跟着扩，节省资源。
- **故障隔离**：Worker 挂了不影响用户刷视频（降级为直写 MySQL），API Server 挂了 Worker 把积压消息消费完就停，不会丢数据。

> **OS 知识点**：进程 vs 线程的选择。Go 项目用 goroutine（用户态轻量线程，M:N 调度），Python 用 asyncio（单线程事件循环）+ 多 worker 进程（uvicorn --workers）。两者都能实现高并发，但模型不同：goroutine 抢占式、asyncio 协作式（需要 await 主动让出执行权）。

#### A.3 为什么选择这些基础设施？

| 组件 | 角色 | 选型理由 | 如果换成... |
|------|------|----------|------------|
| **MySQL 8.0** | 持久化存储 | 关系型、事务保证 ACID、成熟稳定 | PostgreSQL：也行，但 MySQL 在互联网公司更普及 |
| **Redis 7** | 缓存+热榜+限流 | 内存极快、丰富数据结构（ZSET做热榜）、Lua原子脚本 | Memcached：不支持 ZSET，做不了排行榜 |
| **RabbitMQ** | 异步消息 | 可靠投递、死信队列、管理后台成熟 | Kafka：吞吐更高但部署重，RabbitMQ 对这个量级更轻量 |

**关键设计原则：Redis 和 RabbitMQ 都是"可选依赖"**。连不上就自动降级走 MySQL，系统可以用纯 MySQL 跑起来，再加 Redis/RabbitMQ 获得性能提升。这叫 **优雅降级（Graceful Degradation）**。

#### C.2 FastAPI 应用工厂与生命周期（main.py 怎么写）

这是 Python 项目最核心的文件，展示了 FastAPI 的启动/关闭生命周期管理：

```python
# app/main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI
from app.config import settings
from app.database import create_engine, close_engine, create_tables
from app.core.redis import create_redis, close_redis
from app.core.rabbitmq import create_rabbitmq, close_rabbitmq
from app.api.router import api_router


@asynccontextmanager
async def lifespan(app: FastAPI):
    # ========== 启动阶段（Go main.go 做的事） ==========
    # 1. 加载配置（pydantic-settings 自动从 .env / YAML 读取）
    print(f"Config loaded: {settings.model_dump()}")

    # 2. 连接 MySQL（SQLAlchemy async engine）
    await create_engine()
    await create_tables()  # 等价于 GORM AutoMigrate

    # 3. 连接 Redis（可选依赖，连不上不阻塞启动）
    try:
        await create_redis()
        print("Redis connected")
    except Exception:
        print("Redis unavailable, cache disabled")
        app.state.redis = None

    # 4. 连接 RabbitMQ（可选依赖）
    try:
        await create_rabbitmq()
        print("RabbitMQ connected")
    except Exception:
        print("RabbitMQ unavailable, using direct DB writes")
        app.state.rabbitmq = None

    # 5. 启动后台任务（SSE hub、outbox poller）
    # ...

    yield  # ← 应用运行中

    # ========== 关闭阶段（Go defer 做的事） ==========
    await close_rabbitmq()
    await close_redis()
    await close_engine()


app = FastAPI(
    title="短视频Feed流系统",
    version="1.0.0",
    lifespan=lifespan,  # FastAPI 的生命周期钩子
)

app.include_router(api_router, prefix="/api")
```

**关键对比：Go 的 `defer` vs Python 的 `lifespan`**

Go 用 `defer` 在每个资源创建后立即注册清理，Python 用 `@asynccontextmanager` 集中管理。两者效果相同：保证进程退出时资源正确释放（关闭 TCP 连接、刷新缓冲区）。

> **OS 知识点**：无论是 Go 还是 Python，程序退出时必须关闭文件描述符（fd）和 TCP Socket。Linux 每个进程默认最多打开 1024 个 fd（ulimit -n），MySQL/Redis/RabbitMQ 各占一个，HTTP 连接可以复用一个。如果忘记 `defer Close()` 或 `await close()`，在高并发下会耗尽 fd 导致 "too many open files" 错误。

#### C.4 依赖注入：FastAPI Depends 对应 Gin 中间件

这是 Go 和 Python 架构上**最大的差异**，也是面试中一定会被问到的点。

**Go 的方式（Gin 中间件链）**：

```go
// Go: 中间件通过路由分组注入
protectedGroup := r.Group("")
protectedGroup.Use(jwt.JWTAuth(accountRepo, cache))  // 注入鉴权
protectedGroup.POST("/like", likeLimiter, likeHandler.Like)  // 注入限流
```

**Python 的方式（FastAPI Depends 依赖注入）**：

```python
# app/middleware/auth.py
from fastapi import Depends, HTTPException, Request
from fastapi.security import HTTPBearer
from app.core.security import parse_token
from app.core.redis import get_redis

security = HTTPBearer(auto_error=False)

async def get_current_user(
    request: Request,
    credentials = Depends(security),
) -> dict:
    """严格鉴权：不带 token 直接返回 401"""
    if credentials is None:
        raise HTTPException(401, "Missing authorization header")
    claims = parse_token(credentials.credentials)
    if claims is None:
        raise HTTPException(401, "Invalid or expired token")

    # 优先查 Redis 缓存（自愈机制同 Go）
    redis = request.app.state.redis
    if redis:
        cached = await redis.get(f"account:{claims['account_id']}")
        if cached and cached.decode() != credentials.credentials:
            raise HTTPException(401, "Token has been revoked")

    return {"account_id": claims["account_id"], "username": claims["username"]}


async def get_optional_user(
    request: Request,
    credentials = Depends(security),
) -> dict | None:
    """软鉴权：不带 token 不报错，带了必须合法"""
    if credentials is None:
        return None
    claims = parse_token(credentials.credentials)
    if claims is None:
        raise HTTPException(401, "Invalid or expired token")
    return {"account_id": claims["account_id"], "username": claims["username"]}
```

**在路由中使用（依赖注入到函数签名）**：

```python
# app/api/like.py
from fastapi import APIRouter, Depends
from app.middleware.auth import get_current_user
from app.middleware.ratelimit import rate_limit
from app.schemas.like import LikeRequest
from app.services.like import LikeService

router = APIRouter(prefix="/like", tags=["点赞"])

@router.post("/like")
async def like_video(
    req: LikeRequest,                                    # 请求体自动校验
    user: dict = Depends(get_current_user),              # 严格鉴权 → 自动注入
    _: None = Depends(rate_limit("like_write", 30, 60)), # 限流 30次/分钟
    service: LikeService = Depends(get_like_service),    # Service 注入
):
    """
    Go 中 Gin 中间件链做的事：
      protectedLikeGroup.POST("/like", likeLimiter, likeHandler.Like)
    Python 中全部变成了函数参数上的 Depends。
    """
    await service.like(video_id=req.video_id, account_id=user["account_id"])
    return {"message": "like success"}
```

**面试要点：两种注入方式的对比**

| 维度 | Gin 中间件链 | FastAPI Depends |
|------|------------|----------------|
| 注入方式 | `c.Set("accountID", val)` 写 context | 函数参数自动注入 |
| 类型安全 | 运行时 `c.Get("accountID")` 类型断言 | Pydantic 编译时类型检查 |
| 可测试性 | 需要构造 gin.Context | 直接调函数，传 mock 参数 |
| 软鉴权 | 单独写 `SoftJWTAuth` 中间件 | 同一个 Depends，换上 `get_optional_user` |
| 依赖关系 | 中间件之间只能通过 context 传数据 | Depends 可以嵌套：`B(Depends(A))` |

**为什么 FastAPI 的 Depends 更好？** 因为它是声明式的：路由函数签名直接告诉你"这个接口需要什么"——需要登录、需要限流、需要什么 Service。Go 中间件是在路由注册时注入的，要看 router.go 才知道有哪些中间件。

> **OS 知识点**：Gin 和 FastAPI 的并发模型差异。Gin 对每个请求启动一个 goroutine（Go runtime 自动调度到 OS 线程），FastAPI 对每个请求创建一个 asyncio Task（在单个 OS 线程的事件循环中协作调度）。这意味着 Python 中如果 Service 层有同步阻塞操作（如 `time.sleep(5)`），会卡住整个事件循环，所有其他请求都得等。解决办法：`await asyncio.sleep(5)` 或把同步操作放进 `run_in_executor()` 放入线程池。

#### C.5 数据库连接管理：SQLAlchemy async vs GORM

```python
# app/database.py
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker, AsyncSession
from app.models import *  # 导入所有模型，确保 Base 的 metadata 完整

engine = None
SessionLocal = None

async def create_engine():
    global engine, SessionLocal
    url = f"mysql+aiomysql://{user}:{pwd}@{host}:{port}/{db}"
    engine = create_async_engine(url, pool_size=20, max_overflow=40)
    SessionLocal = async_sessionmaker(engine, expire_on_commit=False)

async def get_db() -> AsyncSession:
    """FastAPI Depends 用：每个请求一个独立的数据库会话"""
    async with SessionLocal() as session:
        yield session  # 请求结束时自动 commit 或 rollback

# 在路由中注入：
# @router.post("/xxx")
# async def xxx(db: AsyncSession = Depends(get_db)):
#     ...
```

**Go 的 GORM vs Python 的 SQLAlchemy**：

| 特性 | GORM | SQLAlchemy |
|------|------|-----------|
| 迁移 | `db.AutoMigrate(&Model{})` | Alembic（`alembic revision --autogenerate`） |
| 会话管理 | 所有操作共享 `*gorm.DB` | 每个请求 `AsyncSession`（上下文管理器） |
| 查询语法 | 链式调用 `db.Where().Find()` | `select(Model).where().execute()` |
| 事务 | `db.Transaction(func(tx *gorm.DB){})` | `session.begin()` / `async with session.begin()` |

#### Q1: "你为什么选 Python 重做而不是继续用 Go？"

> 这是一个**动机问题**。回答要点：强调学习目标，不要贬低 Go。

"Go 版本已经是完整实现，我选择用 Python 重做是为了**深入理解后端设计模式**。Go 的 goroutine 和 channel 屏蔽了很多底层细节，Python 的 asyncio 需要我显式管理协程的调度和取消，这反而让我对并发模型理解更深。另外，Go 项目本身就是很优秀的参考实现，我在重写时遵循'先理解 Go 为什么这样做，再用 Python 最地道的方式实现'的原则，而不是逐行翻译。"

#### Q2: "这个系统的 QPS 大概是多少？瓶颈在哪里？"

> 这是一个**性能评估问题**。先分析瓶颈，再给估算。

"单个 API Server 实例在 4C8G 机器上，纯查 Feed 流约 2000-3000 QPS（瓶颈在 Redis），涉及写操作约 500-1000 QPS（瓶颈在 MySQL 事务）。最慢的操作是热度榜查询——需要合并 60 个分钟 ZSET，如果 Redis 内存不足会明显变慢。MySQL 的瓶颈主要在 `ListByFollowing`——需要子查询查出所有关注的人，再 IN 查视频，关注越多越慢。"

#### Q3: "Redis 挂了怎么办？数据会丢吗？"

"Redis 在这个项目中是**纯粹的性能层**，挂了只会影响：Cache miss 全走 MySQL（慢）、热榜无法实时更新、限流失效。核心数据不会丢——因为所有写操作最终都落地 MySQL，Redis 缓存的数据都有 MySQL 作为 source of truth。Redis 恢复后，缓存通过请求自愈——鉴权先查 MySQL 比对通过后回写 Redis。"

#### Q4: "Go 的 singleflight 在 Python 中怎么实现？"

> 这是一个**跨语言映射问题**，考察你是否真正理解了设计模式的本质。

"singleflight 的本质是：**对同一个 key 的并发请求，只让第一个真正执行，其余等待结果**。Python 实现：

```python
import asyncio

class SingleFlight:
    def __init__(self):
        self._inflight: dict[str, asyncio.Event] = {}
        self._results: dict[str, any] = {}

    async def do(self, key: str, fn):
        # 如果这个 key 已经在执行中，等待它的结果
        if key in self._inflight:
            await self._inflight[key].wait()
            return self._results[key]

        # 标记为执行中
        self._inflight[key] = asyncio.Event()
        try:
            result = await fn()
            self._results[key] = result
            return result
        finally:
            self._inflight[key].set()  # 唤醒所有等待者
            # 不要立即清理，给等待者读取结果的时间
```

核心差异：Go 用 `sync.WaitGroup` 阻塞 goroutine，Python 用 `asyncio.Event` 挂起协程。**WaitGroup 是内核级同步原语（基于 futex），Event 是用户态 asyncio 原语**。效果相同，实现机制不同。"

**为什么要分层？面试中这样回答：**

1. **单一职责**：Handler 只管 HTTP，Service 只管业务规则，Repo 只管 SQL。改一个不影响另一个。
2. **可测试**：Service 层 mock 掉 Repo，就可以不连数据库测试业务逻辑。
3. **可替换**：想把 MySQL 换成 PostgreSQL？只改 Repo 层，Service 不动。
4. **与框架解耦**：业务逻辑不依赖 FastAPI/Gin，理论上可以换 Web 框架。

**Worker 层为什么绕过前三层？**

Worker 不走 Handler → Service → Repo 的传统链路，而是直接消费 MQ 消息 → 直接写 Repo。因为 Worker 不需要 HTTP 参数校验、不需要鉴权（消息在 API 层已经鉴权过了），它的唯一职责是"把已经确定正确的事件落地"。这是一种**命令模式**：API 发出命令（LikeTheVideo），Worker 执行命令（INSERT into likes + UPDATE likes_count）。

#### A.2 为什么前端用 Nginx 反向代理而不是 Vite 直连后端？

**设计选择：生产环境加 Nginx，开发环境 Vite 代理**

- **生产环境**：Vue 编译出静态文件后，Nginx 既返回静态资源，又把 `/api` 请求代理到后端。好处：同域访问、Nginx 可以做 gzip/缓存/限流、不需要处理 CORS。
- **开发环境**：Vite dev server 自带 proxy 功能（`vite.config.ts` 中配置），效果等同于 Nginx。

> **计网知识点**：CORS（跨域资源共享）——当浏览器中 `localhost:5173` 的前端向 `localhost:8080` 的后端发请求时，属于不同源（端口不同），浏览器会先发一个 OPTIONS 预检请求。加了 Nginx 代理后 `/api` 请求同域，不需要 CORS，更安全也更简单。

#### A.3 为什么前后端所有接口都用 POST？

Go 参考实现中，从注册、登录到 Feed 查询，**所有接口都是 POST**。这不是 RESTful 风格，而是一个务实的工程选择：

| 选择 | 优点 | 缺点 |
|------|------|------|
| 全 POST（Go 项目做法）| 前端调用统一、参数在 body 里不用拼 URL、不用处理 GET 的 URL 长度限制 | 不符合 REST 语义、浏览器不能缓存 GET 请求 |
| RESTful（GET/POST/PUT/DELETE）| 语义清晰、GET 可被 CDN/浏览器缓存 | 前端要多写几行代码区分请求方法 |

**面试时怎么讲**：Go 项目选了全 POST，Python 重构时可以根据接口性质选择 RESTful——读操作用 GET（`GET /video/{id}`）、写操作用 POST/PUT/DELETE。FastAPI 天然支持 RESTful 路由，用 HTTP 方法区分语义会更规范。

### B. 实现要点（Go 设计思想）

Go 项目中网络通信的核心设计：

1. **容器间通信**：Docker Compose 的 bridge 网络中，容器通过 **服务名**（不是 IP）互相访问。如 MySQL 连接 `mysql:3306`，Redis 连接 `redis:6379`。
2. **健康检查**：`docker-compose.yml` 中每个服务都配了 `healthcheck`，Backend 和 Worker 用 `depends_on: condition: service_healthy` 确保依赖就绪后才启动。
3. **连接重试**：Worker 启动时对 MySQL 和 RabbitMQ 做指数退避重连（1s → 2s → 4s → 8s...最多 30s），因为容器编排中依赖就绪不等于端口已监听。
4. **SSE 通知**：`GET /notification/stream?token=xxx` 使用 HTTP 长连接，响应头 `Content-Type: text/event-stream`，每 30 秒发一次心跳注释 `: keepalive` 防止代理超时断开。

```dockerfile
# Worker 入口是 workers/main.py，不是 app.main:app
CMD ["python", "-m", "app.workers.main"]
```

> **OS 知识点**：`uvicorn --workers 4` 是什么意思？uvicorn 会 fork 出 4 个子进程，每个进程运行一个独立的事件循环。主进程做负载均衡（SO_REUSEPORT），把 TCP 连接分发给子进程。这是 Python 利用多核的标准方式——单进程 asyncio 只能用一个核（GIL 限制），多 worker 进程绕过 GIL。
>
> Worker 进程不用 uvicorn（不监听端口），直接用 `asyncio.run(main())` 启动事件循环。因为它不需要 HTTP Server，只需要消费 RabbitMQ 消息。

#### C.6 连接重试机制（Worker 启动时）

Go 项目中 Worker 有 `connectWithRetry` 函数做指数退避重连。Python 实现：

```python
# app/workers/main.py — Worker 入口
import asyncio
import logging

async def connect_with_retry(name: str, max_retries: int, factory):
    """指数退避重连：1s → 2s → 4s → 8s... 最多30s"""
    for i in range(max_retries):
        try:
            return await factory()
        except Exception as e:
            wait = min(2 ** i, 30)
            logging.warning(f"{name} 不可用，{wait}s 后重试 ({i+1}/{max_retries}): {e}")
            await asyncio.sleep(wait)
    raise RuntimeError(f"{name}: 超过最大重试次数")


async def main():
    # 1. 带重试连 MySQL
    engine = await connect_with_retry("MySQL", 10, create_engine)

    # 2. Redis 和 RabbitMQ 也带重试
    redis = await connect_with_retry("Redis", 10, create_redis)
    rabbitmq = await connect_with_retry("RabbitMQ", 10, create_rabbitmq)

    # 3. 启动所有 Worker（并发消费）
    tasks = [
        asyncio.create_task(like_worker.run()),
        asyncio.create_task(comment_worker.run()),
        asyncio.create_task(social_worker.run()),
        asyncio.create_task(popularity_worker.run()),
        asyncio.create_task(notification_worker.run()),
    ]
    await asyncio.gather(*tasks)


if __name__ == "__main__":
    asyncio.run(main())
```

> **OS 知识点**：`asyncio.gather` vs Go 的 `go func()`。两者效果相同（并发执行多个任务），但错误处理不同。`asyncio.gather` 一个任务抛异常默认会传播到其他任务，需要 `return_exceptions=True` 才能隔离。Go 的 goroutine 更松散——一个 panic 不 recover 会整个进程挂掉。

### D. 面试追问预判

#### Q1: "用户请求从浏览器发出到后端处理完成，中间经过了几层？"

"浏览器的 Fetch/XHR → Vite dev proxy（开发）或 Nginx（生产）→ uvicorn 监听端口 → FastAPI 应用 → 中间件链（CORS → Auth → RateLimit）→ 路由匹配 → Handler 函数 → Service → Repository → SQLAlchemy → MySQL sock 连接。

概括就是：**代理层 → Web服务器层 → 中间件层 → 路由层 → 业务层 → 数据访问层 → 数据库**，共 7 层。每一层都有独立的关注点。"

#### Q2: "Docker 容器之间怎么互相发现？IP 变了怎么办？"

"Docker Compose 的 bridge 网络模式下，每个容器启动时被分配一个内部 IP（如 172.x.x.x），同时 Docker 内置的 DNS 服务器把服务名映射到 IP。容器内写 `mysql:3306` 即可，DNS 自动处理 IP 变化。健康检查保证了容器只有在真正就绪后才对流量开放。"

#### Q3: "SSE 和 WebSocket 的区别？为什么这个项目用 SSE？"

| 维度 | SSE | WebSocket |
|------|-----|-----------|
| 方向 | 单向（服务器→客户端） | 双向 |
| 协议 | HTTP/1.1 长连接 | 独立协议 `ws://` |
| 重连 | 浏览器自动重连 | 需要手写重连逻辑 |
| 代理兼容 | 好（就是 HTTP） | 部分代理不支持 |
| 适用场景 | 通知推送、实时数据流 | 聊天、在线协作 |

"通知场景只需要服务器推送，不需要客户端推给服务器，SSE 更简单、浏览器内置重连、代理兼容性好。"

#### Q4: "如果用户上传一个 200MB 的视频，nginx 和 uvicorn 怎么处理？"

"Nginx 配置了 `client_max_body_size 200M`，超过直接拒绝（413 Payload Too Large），保护后端不会被大文件打爆。通过这个限制后，Nginx 把完整文件代理给 FastAPI。FastAPI 的 `UploadFile` 默认把文件暂存到内存中直到超过一定大小（`spool_max_size`，默认 1MB），然后写入磁盘临时文件。Handler 收到完整文件后移动到永久存储目录。

Python 实现中可以用 FastAPI 的 `UploadFile` 配合 `aiofiles` 做异步流式写入：

```python
@router.post("/uploadVideo")
async def upload_video(file: UploadFile = File(...), user=Depends(get_current_user)):
    # FastAPI 的 UploadFile 已经处理了 multipart/form-data 解析
    # file.size 可做大小校验
    # 流式写入磁盘，不把整个文件加载到内存
    dest = f".run/uploads/videos/{user['account_id']}/{date}/{filename}.mp4"
    async with aiofiles.open(dest, 'wb') as f:
        while chunk := await file.read(1024 * 1024):  # 每次读1MB
            await f.write(chunk)
    return {"url": f"/static/{dest}"}
```

这对比 Go 的 `c.SaveUploadedFile()` 更灵活——可以边读边写、做进度上报、做病毒扫描。"

#### A.2 核心设计原则

**原则一：能冗余就冗余，避免 JOIN**

短视频场景下**读远多于写**，多表 JOIN 代价高。所以 `videos` 表上冗余了 `username`，`comments` 表上冗余了 `username`。代价是改名时要同步更新（触发频率极低），收益是查 Feed 流时不需要 JOIN accounts 表。

> 什么时候不能冗余？数据一致性要求极高的场景（如订单金额），但用户名不一致不会有严重后果。

**原则二：计数用 counter 字段而非 count(*)**

`videos.likes_count` 和 `videos.popularity` 是冗余计数字段。每次点赞不是 `SELECT COUNT(*) FROM likes`，而是直接读这个字段。因为短视频展示频次远高于点赞频次，选存储换读取时间。

> **MySQL 知识点**：`SELECT COUNT(*)` 在 InnoDB 中需要遍历二级索引（或全表扫描），百万级数据时可能几百毫秒。读 counter 字段是 O(1)。

**原则三：关联表用复合唯一索引保证幂等**

`likes(video_id, account_id)` 和 `socials(follower_id, vlogger_id)` 都用了复合唯一索引（UNIQUE INDEX on two columns），既保证了数据不重复，又加速了"是否已点赞"/"是否已关注"的查询。

### B. 设计思想（Go 参考实现提炼）

#### B.1 accounts 表

```sql
-- Go entity: internal/account/entity.go
CREATE TABLE accounts (
    id            BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
    username      VARCHAR(255) NOT NULL UNIQUE,   -- 唯一索引
    password      VARCHAR(255) NOT NULL,           -- bcrypt哈希，JSON序列化时忽略
    token         VARCHAR(512) DEFAULT '',         -- 当前有效JWT
    refresh_token VARCHAR(512) DEFAULT '',         -- 刷新令牌(32字节hex)
    avatar_url    VARCHAR(512) DEFAULT '',         -- 头像URL
    bio           VARCHAR(255) DEFAULT ''          -- 个人简介
);
```

**关键设计点**：
- `password` 存 bcrypt 哈希（60 字符），不是明文。即使数据库泄露攻击者也拿不到密码
- `token` 存在数据库里的原因：JWT 是无状态的，不存 DB 就无法主动失效（如封号）。存了之后可以"校验 token 是否为当前有效 token"
- `refresh_token` 是随机字符串（32 bytes → 64 hex chars），不需要 JWT，因为只在服务端使用

#### B.2 videos 表（索引最多的表）

```sql
-- Go entity: internal/video/video_entity.go
CREATE TABLE videos (
    id          BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
    author_id   BIGINT UNSIGNED NOT NULL,
    username    VARCHAR(255) NOT NULL,             -- 冗余，避免JOIN
    title       VARCHAR(255) NOT NULL,
    description VARCHAR(255) DEFAULT '',
    play_url    VARCHAR(512) NOT NULL,             -- 视频文件路径
    cover_url   VARCHAR(512) NOT NULL,             -- 封面图路径
    create_time DATETIME(3) NOT NULL,
    likes_count BIGINT NOT NULL DEFAULT 0,
    popularity  BIGINT NOT NULL DEFAULT 0,

    -- 三个关键索引：
    INDEX idx_create_time (create_time DESC),                          -- 最新Feed流
    INDEX idx_likes_count_id (likes_count DESC, id DESC),              -- 点赞排行分页
    INDEX idx_popularity_time_id (popularity DESC, create_time DESC, id DESC)  -- 热度榜分页
);
```

**索引设计分析**：

| 索引 | 服务哪个接口 | 为什么这样排 |
|------|-------------|-------------|
| `(create_time DESC)` | `/feed/listLatest` 游标分页 | `WHERE create_time < ?` 只用一个字段 |
| `(likes_count DESC, id DESC)` | `/feed/listLikesCount` 复合游标 | likes_count 可重复，加 id 保证稳定排序 |
| `(popularity DESC, create_time DESC, id DESC)` | `/feed/listByPopularity` 三字段游标 | popularity 和 create_time 都可能重复 |

> **MySQL 核心知识点 — 复合索引的最左前缀原则**：
> `INDEX (a, b, c)` 可以加速以下查询：
> - `WHERE a = 1` （用到 a）
> - `WHERE a = 1 AND b = 2` （用到 a, b）
> - `WHERE a = 1 AND c = 3` （只用到 a，跳过了 b 所以 c 用不上！）
> - `ORDER BY a, b, c` （用到全部）
>
> **不能加速** `WHERE b = 2`（跳过了 a），`ORDER BY c`（跳过了 a, b）。
>
> 所以我们在 Feed 查询中用的是 `WHERE (popularity < ?) OR (popularity = ? AND create_time < ?)` ——完全匹配了索引的列顺序。

#### B.3 likes 表

```sql
-- Go entity: internal/video/like_entity.go
CREATE TABLE likes (
    id         BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
    video_id   BIGINT UNSIGNED NOT NULL,
    account_id BIGINT UNSIGNED NOT NULL,
```

likes 表只记录"谁 -> 点了 -> 哪个视频"，计数冗余在 videos 表。因为读 Feed 流时需要展示每个视频的点赞数，如果每次都 `COUNT(*) FROM likes WHERE video_id = ?` 会很慢。

#### B.7 messages 表（私信）

```sql
CREATE TABLE messages (

    is_read    BOOLEAN DEFAULT FALSE,

    INDEX idx_to_from (to_id, from_id)    -- 查"他发给我的聊天记录"

```

双向查询索引：私信列表需要查 `(from_id=A, to_id=B) OR (from_id=B, to_id=A)`，两个复合索引分别覆盖两个方向的查询。

#### B.8 outbox_msgs 表（事务发件箱 · 基础设施）

```sql
CREATE TABLE outbox_msgs (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    video_id    BIGINT NOT NULL,
    event_type  VARCHAR(50) NOT NULL,
    create_time DATETIME NOT NULL,
    status      VARCHAR(50) NOT NULL DEFAULT 'pending',  -- pending → processed
    INDEX idx_status (status)
);
```

这是保证 MQ 消息可靠投递的关键：视频发布和发件箱记录写在同一事务中，OutboxWorker 轮询 status=pending 的行 → 投递 MQ → 标记 processed。即使 MQ 宕机，消息也不会丢（存在 MySQL 里等着重试）。

```sql
    target_id    BIGINT NOT NULL,       -- 关联对象 ID
```

#### C.4 Repository 层的 SQLAlchemy 写法

Go 用 GORM 链式调用，Python 用 SQLAlchemy 2.0 风格的 `select()` + `async session`：

```python
# app/repositories/feed.py
from sqlalchemy import select, and_, or_
from sqlalchemy.ext.asyncio import AsyncSession
from app.models.video import Video

class FeedRepository:
    def __init__(self, session: AsyncSession):
        self.session = session

    async def list_latest(self, limit: int, latest_before: datetime | None) -> list[Video]:
        """游标分页：按 create_time 降序"""
        stmt = select(Video).order_by(Video.create_time.desc())
        if latest_before:
            stmt = stmt.where(Video.create_time < latest_before)
        stmt = stmt.limit(limit)
        result = await self.session.execute(stmt)
        return list(result.scalars().all())

    async def list_likes_count(self, limit: int, cursor: tuple[int, int] | None) -> list[Video]:
        """复合游标分页：(likes_count, id)"""
        stmt = select(Video).order_by(Video.likes_count.desc(), Video.id.desc())
        if cursor:
            likes_before, id_before = cursor
            stmt = stmt.where(
                or_(
                    Video.likes_count < likes_before,
                    and_(Video.likes_count == likes_before, Video.id < id_before),
                )
            )
        stmt = stmt.limit(limit)
        result = await self.session.execute(stmt)
        return list(result.scalars().all())
```

> **安全知识点 — SQL 注入防护**：
> GORM 和 SQLAlchemy 都用参数化查询（parameterized query），把用户输入和 SQL 结构分离。
> 看 Go 代码中的 `db.Where("username = ?", username)` 和 Python 的 `select(Video).where(Video.username == username)`——它们都不会拼接字符串，数据库驱动会把参数单独发送，永远不会把用户输入当成 SQL 执行。

### D. 面试追问预判

#### Q1: "如果说点赞数不需要实时精确，你还能怎么设计？"

"可以异步批处理：先把点赞写入 Redis `SET video:{id}:likes` + `SADD`，每 10 秒一个定时任务把增量 `INCRBY` 到 MySQL。Redis 挂了丢 10 秒内的计数（可接受），但响应速度提升 10 倍以上。也可以写到 RabbitMQ，Worker 批量更新 videos 表，减少单条 UPDATE 对 MySQL 的争抢。"

#### Q2: "inotifications 表数据量大了怎么办？"

"冷热分离：保留最近 30 天的通知在主表，超过 30 天的归档到历史表或对象存储。查询时优先查主表，如果没结果或用户明确要看历史，再去归档表查。更进一步可以分表——按 recipient_id 哈希分 16 张表。"

#### Q3: "如果 username 修改了，videos 和 comments 里的冗余 username 怎么更新？"

"有三种策略：
1. **同步更新**：改名的同时 `UPDATE videos SET username=? WHERE author_id=?`（MySQL 事务保证）。简单但改名很慢。
2. **异步更新**：发 MQ 消息，Worker 慢慢更新。改名立刻返回，但短暂时间内新旧名字共存。
3. **不冗余**：JOIN accounts 查。性能差但数据最一致。

Go 项目用的第 2 种（Rename 操作更新 account 表 + 发新 token，冗余字段通过后续逻辑自然更新）。面试中关键要讲出三种方案的 trade-off。"

#### Q4: "复合索引 (A, B) 和两个单列索引 A + B 的区别？"

"复合索引 `(A, B)` 形成一棵 B+ 树，先按 A 排序，A 相同时按 B 排序。两个单列索引是两棵独立的 B+ 树。
- `WHERE A = 1 AND B = 2`：复合索引一次定位；单列索引需要先查 A 的 B+ 树再查 B 的 B+ 树，然后取交集（index merge），效率低。
- `WHERE B = 2`：复合索引用不上（最左前缀），单列索引 B 可以用。
- 存储空间：复合索引 1 棵 B+ 树，两个单列索引 2 棵 B+ 树。

我们这个项目用复合索引是因为查询模式固定（总是按 `likes_count + id` 或 `popularity + create_time + id` 排序），不涉及单独查第二个字段。"

#### Q5: "为什么要用 BIGINT 而不是 INT？"

"INT 最大值约 21 亿。对于一个小型项目是够的，但如果你不知道将来会有多少数据，BIGINT（约 922 京）几乎等于无限。多花 4 字节的存储换永远不会溢出，是合理的保险。真正有区分度的场景是分库分表做 sharding 时，INT 范围可能不够分。"

### A. 设计决策

#### A.1 双 Token 模型：为什么需要两个 Token？

```
Access Token (JWT, 15分钟)          Refresh Token (随机串, 7天)
┌─────────────────────┐            ┌─────────────────────┐
│ 携带用户身份信息      │            │ 仅用于换新的Access   │
│ 短有效期，泄露影响小   │            │ 长有效期，存在DB里    │
│ 每次请求都带          │            │ 只在刷新时用         │
│ 无状态，不解密即可验证 │            │ 有状态，可以主动失效   │
└─────────────────────┘            └─────────────────────┘
```

**为什么不只用一个长有效期的 JWT？** 如果只用一个 7 天有效的 JWT，一旦泄露（如中间人攻击、日志泄露），攻击者可以冒充用户 7 天。双 Token 把风险隔离：Access Token 泄露影响只有 15 分钟，Refresh Token 不随请求发送（只在 `/refresh` 接口用），泄露面小得多。

**为什么 Access Token 是 JWT 而 Refresh Token 是随机字符串？**
- Access Token 用 JWT：每次请求都要验证，JWT 可以不解密直接验签名（O(1)），不需要查 DB
- Refresh Token 用随机串：只有 `/refresh` 一个接口用，必须查 DB 比对，不需要 JWT 的自解释能力

> **安全知识点**：JWT 的三个部分——Header（算法类型）、Payload（用户数据，Base64 编码，**不加密只编码**）、Signature（签名）。验证过程：用密钥对 Header+Payload 重新计算签名，比对是否一致。如果 Payload 被篡改，签名不匹配，验证失败。所以 JWT 的 Payload 不要放敏感信息（如密码），Base64 编码 = 明文。

#### A.2 Token 为什么存在数据库里？

JWT 是无状态的——拿到 Token 就能验证身份，不需要查 DB。但这也意味着**无法主动失效**。解决方法：在 accounts 表存 `token` 字段（当前有效 Access Token），鉴权时比对请求中的 Token 和 DB 中的 Token 是否一致。不一致说明 Token 已被替换（用户改了密码 / 登出了 / 在其他设备登录了），拒绝请求。

```
请求中的Token = "eyJhbG..."
DB中的Token   = "eyJhbG..."  → 一致 → 通过
                             → 不一致 → 401 (Token已被失效)
```

#### A.3 Redis 在鉴权中的角色：不是必须，但能加速

```
鉴权流程：
  ┌─ 优先查 Redis: account:＜id＞ 是否存在且值=请求Token
  │   → 命中：直接通过（1次 Redis 网络往返，约 0.5ms）
  │   → 未命中/Redis挂了 ↓
  ├─ 降级查 MySQL: SELECT token FROM accounts WHERE id=?
  │   → 通过后回写 Redis（自愈机制：Redis恢复后自动热起来）
  └─ 不匹配 → 401
```

### B. 设计思想（Go 参考实现提炼）

Go 项目用 Gin 中间件实现了两套鉴权：

1. **严格鉴权（JWTAuth）**：不带 Token 或 Token 无效 → 直接 401。保护接口：点赞/评论/关注/发视频/改资料等
2. **软鉴权（SoftJWTAuth）**：带了就必须合法，不带也可以（user = None）。保护接口：Feed 流（匿名用户可以刷，登录用户可看到点赞状态）

中间件验证通过后，把 `accountID` 和 `username` 存入 Gin Context（`c.Set("accountID", val)`），后续 Handler 通过 `c.Get("accountID")` 取用。

### C. Python + FastAPI 实现细节（重点，含详细注释）

#### C.1 JWT 核心工具模块

```python
# app/core/security.py
"""
鉴权核心模块：JWT 签发/验证、bcrypt 密码哈希、Refresh Token 生成。
对应 Go 项目中的 internal/auth/jwt.go 和 internal/account/service.go 中的密码处理。
"""

import os
import secrets
from datetime import datetime, timedelta, timezone
import jwt  # PyJWT 库
from passlib.context import CryptContext  # bcrypt 封装


# ============================================================
# 密码哈希 — 使用 bcrypt
# ============================================================
# passlib 的 CryptContext 封装了 bcrypt 的加盐和比对，
# schemes=["bcrypt"] 指定算法，deprecated="auto" 自动处理算法升级
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")


def hash_password(plain: str) -> str:
    """将明文密码哈希为 bcrypt 密文（60字符，含随机盐）"""
    return pwd_context.hash(plain)


def verify_password(plain: str, hashed: str) -> bool:
    """验证密码是否匹配。注意：这是 CPU 密集型操作（bcrypt 故意慢，防暴力破解）"""
    return pwd_context.verify(plain, hashed)


# ============================================================
# JWT 签发与验证 — HS256 对称加密
# ============================================================
# HS256：HMAC with SHA-256，用一个密钥同时签名和验证
# RS256：RSA 非对称，私钥签名公钥验证（微服务间常用，这里 HS256 就够了）

def _get_jwt_secret() -> str:
    """
    从环境变量读取 JWT 密钥。生产环境务必设置 JWT_SECRET 环境变量！
    如果没设置，生成一个随机密钥（服务重启后所有旧 Token 失效）。
    """
    secret = os.getenv("JWT_SECRET")
    if not secret:
        # 开发环境降级：生成随机 64 字符密钥
        secret = secrets.token_hex(32)
        print("WARNING: JWT_SECRET not set, using random key. All tokens invalid on restart.")
    return secret


def create_access_token(account_id: int, username: str) -> str:
    """
    签发 Access Token（JWT, HS256, 15分钟有效）。

    Payload 结构（对应 Go 的 auth.Claims）：
      {
        "account_id": 1,
        "username": "zhangsan",
        "exp": 1715000000,    # 过期时间（UNIX时间戳）
        "iat": 1714999100,    # 签发时间
        "nbf": 1714999100     # 生效时间（在此之前不可用）
      }
    """
    now = datetime.now(timezone.utc)
    payload = {
        "account_id": account_id,
        "username": username,
        "exp": now + timedelta(minutes=15),   # 15分钟后过期
        "iat": now,                            # 签发时间
        "nbf": now,                            # 立即生效
    }
    token = jwt.encode(payload, _get_jwt_secret(), algorithm="HS256")
    return token


def decode_access_token(token: str) -> dict | None:
    """
    验证并解码 Access Token。
    返回 None 表示 Token 无效（过期、签名错误、被篡改等）。
    注意：不查数据库，纯本地验证——这是 JWT 的最大优势。
    """
    try:
        payload = jwt.decode(
            token,
            _get_jwt_secret(),
            algorithms=["HS256"],
            options={"require": ["exp", "iat"]},  # 要求必须有 exp 和 iat
        )
        return payload
    except jwt.ExpiredSignatureError:
        # Token 过期 — 前端应该用 Refresh Token 换新的
        return None
    except jwt.InvalidTokenError:
        # 签名错误 / 格式错误 / 篡改
        return None


def create_refresh_token() -> str:
    """
    生成 Refresh Token：32 字节随机数 → 64 字符 hex 字符串。
    不需要 JWT，因为只在服务端使用（存在 accounts.refresh_token 字段里）。
    对比 Go 的 auth.GenerateRefreshToken()。
    """
    return secrets.token_hex(32)
```

#### C.2 鉴权依赖注入（FastAPI Depends）

```python
# app/middleware/auth.py
"""
鉴权中间件 — 用 FastAPI Depends 实现，比 Gin 中间件链更显式。
每个需要鉴权的接口在函数参数中声明 Depends(get_current_user) 即可。

对比 Go 的 internal/middleware/jwt/jwt.go：
  Go 中间件通过 c.Set("accountID", val) 隐式传递用户信息
  Python Depends 通过函数返回值显式传递用户信息，类型安全
"""

from typing import Optional
from fastapi import Depends, HTTPException, status, Request
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
from sqlalchemy.ext.asyncio import AsyncSession
from app.database import get_db
from app.core.security import decode_access_token
from app.core.redis import get_redis_client


# FastAPI 内置的 Bearer Token 提取器
# auto_error=False：当请求没有 Authorization 头时不自动报错，而是返回 None
bearer_scheme = HTTPBearer(auto_error=False)


# ============================================================
# 严格鉴权：不带 Token → 401
# 用于：点赞/评论/关注/发视频/改资料/登出 等接口
# ============================================================
async def get_current_user(
    request: Request,
    credentials: Optional[HTTPAuthorizationCredentials] = Depends(bearer_scheme),
    db: AsyncSession = Depends(get_db),
) -> dict:
    """
    严格鉴权依赖。用法：在路由函数参数中写 user: dict = Depends(get_current_user)。

    验证流程（对应 Go 的 JWTAuth 中间件）：
      1. 提取 Authorization: Bearer <token>
      2. 本地验证 JWT 签名 + 有效期（不查 DB）
      3. 优先查 Redis account:<id> 是否有此 Token（自愈机制）
      4. Redis 未命中则查 MySQL accounts.token 比对
      5. 不匹配 → 401（Token 已被失效）
      6. 通过 → 回写 Redis → 返回 {account_id, username}
    """
    # 步骤 1：提取 Token
    if credentials is None:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="缺少 Authorization 头，请先登录",
        )

    token = credentials.credentials

    # 步骤 2：本地验证 JWT（不查 DB，利用 JWT 的自解释特性）
    payload = decode_access_token(token)
    if payload is None:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Token 无效或已过期，请重新登录",
        )

    account_id = payload["account_id"]
    username = payload["username"]

    # 步骤 3：优先查 Redis — 是否存在此 Token（最快的校验路径）
    redis = get_redis_client(request)
    if redis is not None:
        cached = await redis.get(f"v1:account:{account_id}")
        if cached is not None:
            # Redis 中有这个 Key：比对 Token 是否匹配
            if cached.decode("utf-8") == token:
                return {"account_id": account_id, "username": username}
            else:
                # Token 不匹配 → 说明该用户的 Token 已被替换（改名/改密/登出）
                raise HTTPException(
                    status_code=status.HTTP_401_UNAUTHORIZED,
                    detail="Token 已失效，请重新登录",
                )
        # cached is None → Redis 中没这个 Key，继续查 MySQL

    # 步骤 4：降级查 MySQL（Redis 挂了或 key 过期时走这里）
    from app.repositories.account import AccountRepository
    repo = AccountRepository(db)
    account = await repo.find_by_id(account_id)

    if account is None or account.token == "" or account.token != token:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Token 已失效，请重新登录",
        )

    # 步骤 5：自愈回写 Redis（Redis 恢复后通过正常请求自动热起来）
    if redis is not None:
        try:
            await redis.set(f"v1:account:{account_id}", token, ex=86400)  # TTL 24小时
        except Exception:
            pass  # 回写失败不影响业务，下次请求继续走 MySQL

    return {"account_id": account_id, "username": username}


# ============================================================
# 软鉴权：不带 Token 不报错，带了必须合法
# 用于：Feed 流（匿名可刷，登录可看点赞状态）
# ============================================================
async def get_optional_user(
    request: Request,
    credentials: Optional[HTTPAuthorizationCredentials] = Depends(bearer_scheme),
    db: AsyncSession = Depends(get_db),
) -> Optional[dict]:
    """
    软鉴权依赖（对应 Go 的 SoftJWTAuth）。
    用法：user: Optional[dict] = Depends(get_optional_user)
    
    与严格鉴权的区别：
      - 不带 Token → 返回 None（不报错，允许匿名访问）
      - 带了但无效 → 仍然 401（带了就必须合法）
    """
    if credentials is None:
        # 没带 Token，返回 None（匿名用户）
        return None

    # 带了 Token，走严格验证
    token = credentials.credentials
    payload = decode_access_token(token)
    if payload is None:
        # 带了但无效 → 401（不能让不合法 Token 混过去）
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Token 无效或已过期",
        )

    account_id = payload["account_id"]

    # Redis 快速路径（同严格鉴权）
    redis = get_redis_client(request)
    if redis is not None:
        cached = await redis.get(f"v1:account:{account_id}")
        if cached is not None:
            if cached.decode("utf-8") == token:
                return {"account_id": account_id, "username": payload["username"]}
            raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Token 已失效")

    # MySQL 降级路径
    from app.repositories.account import AccountRepository
    repo = AccountRepository(db)
    account = await repo.find_by_id(account_id)
    if account is None or account.token == "" or account.token != token:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="Token 已失效")

    # 自愈回写
    if redis is not None:
        try:
            await redis.set(f"v1:account:{account_id}", token, ex=86400)
        except Exception:
            pass

    return {"account_id": account_id, "username": payload["username"]}
```

```python
class AccountService:
    def __init__(self, repo: AccountRepository, redis: RedisClient = None):
        self.repo = repo
        self.redis = redis

    async def login(self, username: str, password: str) -> LoginResponse:
        """
        登录流程（对应 Go 的 account/service.go:Login）：
          1. 查用户 → 2. 验密码 → 3. 签发双Token → 4. 存MySQL → 5. 写Redis
        """
        # 1. 查用户
        account = await self.repo.find_by_username(username)
        if account is None:
            raise HTTPException(401, "用户名或密码错误")

        # 2. 验密码（bcrypt.verify — 这是 CPU 密集型操作）
        if not verify_password(password, account.password):
            raise HTTPException(401, "用户名或密码错误")  # 不透露是哪个错了

        # 3. 签发双Token
        access_token = create_access_token(account.id, account.username)
        refresh_token = create_refresh_token()

        # 4. 存入 MySQL：同时更新 token 和 refresh_token
        await self.repo.update_tokens(account.id, access_token, refresh_token)

        # 5. 写入 Redis 缓存（可选依赖，写失败不影响返回）
        if self.redis:
            try:
                # 缓存 Access Token：key=account:<id>, value=token, TTL=24h
                await self.redis.set(f"v1:account:{account.id}", access_token, ex=86400)
                # 缓存 Refresh Token：key=refresh:<token>, value=account_id, TTL=7d
                await self.redis.set(f"v1:refresh:{refresh_token}", str(account.id), ex=604800)
            except Exception:
                pass  # Redis 写失败不影响登录

        return LoginResponse(
            token=access_token,
            refresh_token=refresh_token,
            account_id=account.id,
            username=account.username,
        )

    async def refresh_access_token(self, refresh_token: str) -> str:
        """
        用 Refresh Token 换新的 Access Token。
        流程：
          1. 优先查 Redis refresh:<token> → 获取 account_id
          2. Redis 未命中 → 遍历 accounts 表找匹配的 refresh_token
          3. 签发新 Access Token → 更新 DB → 更新 Redis
        """
        account_id = None

        # 快速路径：Redis 查找
        if self.redis:
            try:
                cached = await self.redis.get(f"v1:refresh:{refresh_token}")
                if cached:
                    account_id = int(cached)
            except Exception:
                pass

        # 慢路径：遍历 DB 查找
        if account_id is None:
            account = await self.repo.find_by_refresh_token(refresh_token)
            if account is None:
                raise HTTPException(401, "Refresh Token 无效")
            account_id = account.id

        # 签发新 Access Token
        account = await self.repo.find_by_id(account_id)
        new_token = create_access_token(account.id, account.username)

        # 持久化
        await self.repo.update_token(account_id, new_token)
        if self.redis:
            try:
                await self.redis.set(f"v1:account:{account_id}", new_token, ex=86400)
            except Exception:
                pass

        return new_token

    async def logout(self, account_id: int):
        """
        登出：清空 MySQL 中的 token + refresh_token，删除 Redis 缓存。
        旧 Token 立即失效，任何地方再使用都会 401。
        """
        await self.repo.clear_tokens(account_id)

        if self.redis:
            try:
                await self.redis.delete(f"v1:account:{account_id}")
                await self.redis.delete(f"v1:account:{account_id}:refresh")
            except Exception:
                pass
```

### D. 面试追问预判

#### Q1: "JWT 怎么防止被篡改？"

"JWT 的第三部分是签名（Signature），它是用密钥对 Header+Payload 的 HMAC-SHA256 哈希。客户端收到 Token 后如果修改了 Payload（比如把 account_id 从 1 改成 2），签名就不匹配了。因为攻击者没有密钥，无法重新计算合法签名。服务端验证时发现签名不匹配，直接拒绝。"

#### Q2: "如果我想让某个用户立即下线，怎么做？"

"改他的 Token 即可——把 MySQL 的 accounts.token 字段更新或清空。下次该用户请求时，鉴权中间件发现请求中的 Token 和 DB 中不一致，返回 401。这就是 JWT 存 DB 的原因——不存 DB 就无法主动失效。"

#### Q3: "Refresh Token 泄露了怎么办？"

"Refresh Token 泄露比 Access Token 泄露更严重（有效期 7 天 vs 15 分钟）。防护措施：
1. Refresh Token 只在 `/refresh` 接口用一次，不携带在每个请求中，泄露面小
2. 可以实现 Token 轮转（Refresh Token Rotation）：每次换 Access Token 时同时换一个新的 Refresh Token，旧 Refresh Token 立即失效。这样如果 Refresh Token 泄露，真用户下次刷新时用了旧 Token → 服务端发现不一致 → 知道泄露了 → 清掉所有 Token 强制重新登录
3. 设备指纹 + IP 监控：Refresh Token 被从不常用 IP 使用时告警"

#### Q4: "bcrypt 为什么故意设计得很慢？"

"bcrypt 的 verify 操作大约需要 200-300ms，这是故意的。目的是增加暴力破解的成本——攻击者每秒只能尝试 3-5 个密码（单核），而用 SHA256 可以每秒尝试数百万个。如果数据库泄露，bcrypt 能让攻击者几乎不可能从哈希反推出弱密码。

> **OS 知识点**：bcrypt 消耗的主要是 CPU 时间（迭代哈希计算），在 asyncio 中这是一个同步阻塞操作。如果直接在协程中调用 `bcrypt.verify()`，会阻塞整个事件循环。解决方案：用 `run_in_executor()` 将 bcrypt 操作放到线程池中执行，不阻塞主事件循环。"

```python
import asyncio

async def verify_password_async(plain: str, hashed: str) -> bool:
    """在线程池中执行 bcrypt 验证，不阻塞事件循环"""
    loop = asyncio.get_running_loop()
    # None 参数表示使用默认的 ThreadPoolExecutor
    return await loop.run_in_executor(None, pwd_context.verify, plain, hashed)
```

### A. 设计决策

#### A.1 四种核心写操作的共性模式

```
视频发布  →  事务包三件事（INSERT视频 + INSERT发件箱 + INSERT标签）
点赞     →  异步优先 + 降级兜底（MQ → MySQL双路径）
评论     →  同点赞 + @mention提取通知
关注     →  同点赞（校验更严格：禁止自关 + 防重复）
```

**核心设计模式：异步优先 + 降级兜底**

这是整个项目最有教学价值的设计模式。它的本质是：

```
正常路径（快速）：  API → 发MQ消息 → 立即返回200  (1次网络往返)
                    ↓
                  Worker异步消费 → 写MySQL/Redis

异常路径（保底）：  MQ发送失败 → 降级为直接写MySQL事务  (2-3次网络往返，慢但可靠)
```

**面试金句**：MQ 是加速路径，MySQL 是保底路径。快可以降级，但数据不能丢。

#### A.2 为什么视频发布用了不同的模式（事务发件箱）？

视频发布不能用简单的"MQ 优先 + 降级"模式，因为它需要**同时做三件事**：

1. INSERT 视频记录到 videos 表
2. 通知关注者（需要发 MQ 消息到 timeline）
3. 提取 #话题标签写入 tags + video_tags 表

如果 1 和 3 成功了但 2 失败了（MQ 消息没发出去），关注者就看不到新视频。解决方案：**事务发件箱模式（Transactional Outbox）**——把 MQ 消息也写进同一个 MySQL 事务中，由一个后台 Poller 轮询投递。

```
BEGIN TRANSACTION
  INSERT INTO videos (...)
  INSERT INTO outbox_msgs (video_id, event_type="publish", status="pending")
  INSERT INTO video_tags (...)  -- 提取的 #话题标签
COMMIT
         ↓
OutboxWorker 轮询: SELECT * FROM outbox_msgs WHERE status='pending'
         ↓
投递 timeline MQ → 更新 status='processed'
```

### B. 设计思想（Go 参考实现提炼）

**视频发布**：`video_service.go:Publish()` 中，一个 GORM 事务包裹了 Create(video) + Create(outboxMsg) + 标签提取写入。事务完成后，outbox poller 异步投递 MQ，失败也不影响已发布的数据。

```python
# app/services/video.py
"""
视频发布 Service — 展示了事务发件箱模式。
对应 Go 的 internal/video/video_service.go:Publish()
"""

import re
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select
from app.models.video import Video
from app.models.outbox import OutboxMsg
from app.models.tag import Tag, VideoTag


# 提取 #话题 的正则：匹配 #后跟Unicode字母/数字/下划线
# 例如："今天天气真好 #旅行 #美食推荐" → ["旅行", "美食推荐"]
TAG_PATTERN = re.compile(r'#([\p{L}\p{N}_]+)')


def extract_tags(text: str) -> list[str]:
    """从文本中提取 #话题标签，去重"""
    matches = TAG_PATTERN.findall(text)
    return list(dict.fromkeys(matches))  # 去重并保持顺序


async def publish_video(
    session: AsyncSession,  # 这个session来自Depends(get_db)
    author_id: int,
    username: str,
    title: str,
    description: str,
    play_url: str,
    cover_url: str,
) -> Video:
    """
    发布视频 — 一个事务完成三件事。
    
    流程：
      1. INSERT video
      2. INSERT outbox_msg (pending状态)
      3. 提取 #话题 → INSERT/UPDATE tags → INSERT video_tags
      这三个操作在同一个事务中，要么全部成功，要么全部回滚。
    """
    # 参数校验
    title = title.strip()
    if not title:
        raise ValueError("标题不能为空")
    if not play_url or not cover_url:
        raise ValueError("视频文件和封面不能为空")

    # ★ 开启事务（async with session.begin() 自动 commit/rollback）
    async with session.begin():
        # ---- 第1步：创建视频记录 ----
        video = Video(
            author_id=author_id,
            username=username,         # 冗余，避免JOIN
            title=title,
            description=description.strip(),
            play_url=play_url,
            cover_url=cover_url,
            # create_time 由数据库 DEFAULT CURRENT_TIMESTAMP 自动填充
            likes_count=0,
            popularity=0,
        )
        session.add(video)
        await session.flush()  # flush 让 video.id 生成出来，后续步骤需要它

        # ---- 第2步：写入发件箱（保证 MQ 消息不丢） ----
        outbox = OutboxMsg(
            video_id=video.id,
            event_type="video_published",
            status="pending",
        )
        session.add(outbox)

        # ---- 第3步：提取并写入 #话题标签 ----
        tags_text = extract_tags(f"{title} {description}")
        for tag_name in tags_text:
            # 3a. 查找已有标签，不存在则创建（FirstOrCreate 模式）
            result = await session.execute(
                select(Tag).where(Tag.name == tag_name)
            )
            tag = result.scalar_one_or_none()
            if tag is None:
                tag = Tag(name=tag_name)
                session.add(tag)
                await session.flush()  # 获取 tag.id

            # 3b. 创建视频-标签关联
            session.add(VideoTag(video_id=video.id, tag_id=tag.id))

    # 事务在此处自动 commit（async with 块结束时）
    # 如果任何一步抛异常，session.begin() 上下文管理器自动 rollback

    return video
```

#### C.2 OutboxWorker：轮询发件箱投递 MQ

```python
# app/workers/outbox_worker.py
"""
事务发件箱轮询器 — 后台任务，持续扫描 outbox_msgs 表中的 pending 记录，
投递到 RabbitMQ Timeline Exchange，成功后标记为 processed。

对应 Go 的 internal/worker/outboxworker.go
"""

import asyncio
import logging
from sqlalchemy import select, delete
from app.database import SessionLocal
from app.models.outbox import OutboxMsg
from app.core.rabbitmq import get_timeline_producer


async def run_outbox_poller(poll_interval: float = 1.0):
    """
    Outbox Poller 主循环。
    每隔 poll_interval 秒扫描一次 outbox_msgs 表，投递 pending 消息。
    """
    producer = get_timeline_producer()
    if producer is None:
        logging.warning("Outbox poller disabled: RabbitMQ not available")
        return

    while True:
        try:
            async with SessionLocal() as session:
                # 每次取 100 条 pending 消息（按时间升序，先进先出）
                result = await session.execute(
                    select(OutboxMsg)
                    .where(OutboxMsg.status == "pending")
                    .order_by(OutboxMsg.create_time.asc())
                    .limit(100)
                )
                messages = result.scalars().all()

                for msg in messages:
                    try:
                        # 投递到 RabbitMQ Timeline Exchange
                        await producer.publish(
                            video_id=msg.video_id,
                            create_time=msg.create_time,
                        )
                        # 投递成功 → 标记为 processed（这里用删除，省空间）
                        await session.execute(
                            delete(OutboxMsg).where(OutboxMsg.id == msg.id)
                        )
                        await session.commit()
                    except Exception as e:
                        logging.error(f"Outbox 投递失败 video_id={msg.video_id}: {e}")
                        # 失败不删除，下次循环重试
                        await session.rollback()

        except Exception as e:
            logging.error(f"Outbox poller 异常: {e}")

        await asyncio.sleep(poll_interval)
```

#### C.3 点赞/取消点赞：异步优先 + 降级兜底（⭐ 最重要）

这是整个项目**最核心的设计模式**，面试中一定会被重点追问。

```python
# app/services/like.py
"""
点赞 Service — 展示了异步优先 + 降级兜底模式。
对应 Go 的 internal/video/like_service.go:Like() 和 Unlike()

核心思想：
  正常情况：发MQ消息 → 立即返回 (快)
  MQ故障时：降级为直接写MySQL事务 (慢但可靠)
  两条路径保证：点赞数据一定落地，差别只在响应速度
"""

import logging
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select, update
from app.models.video import Video
from app.models.like import Like


async def like_video(
    session: AsyncSession,
    video_id: int,
    account_id: int,
    like_mq,          # LikeMQ 生产者（可能是 None，表示 MQ 不可用）
    popularity_mq,    # PopularityMQ 生产者（可能是 None）
    redis_client,     # Redis 客户端（可能是 None）
) -> None:
    """
    点赞 — 异步优先 + 降级兜底。

    链路追踪（正常路径）：
      用户点❤️ → POST /like/like → Handler → Service.like_video()
        → 校验视频存在 + 未重复点赞
        → likeMQ.like() 发送消息到 like.events   ─┐
        → popularityMQ.update(+1) 发送到 popularity.events ─┤ 这两条并发发
        → 两条都成功 → 立即返回 200                     ←┘
        → LikeWorker 异步消费 → INSERT likes + UPDATE likes_count
        → PopularityWorker 异步消费 → ZINCRBY Redis 热榜

    链路追踪（MQ 故障降级路径）：
      likeMQ.like() 失败！
        → 不慌，降级为 MySQL 事务直写：
          BEGIN
            SELECT id FROM videos WHERE id=? (再确认视频存在)
            INSERT INTO likes (video_id, account_id)     ← 如果重复→1062错误
            UPDATE videos SET likes_count = likes_count + 1
            UPDATE videos SET popularity = popularity + 1
          COMMIT
        → 返回 200

    链路追踪（MQ 一半成功一半失败的混合路径）：
      likeMQ 成功，popularityMQ 失败！
        → MySQL 降级写入不执行（like 数据由 Worker 落地）
        → 直接调 update_popularity_cache() 写 Redis ZSET
        → 返回 200
    """

    # ===== 第1步：参数校验 + 业务规则检查 =====
    if video_id <= 0 or account_id <= 0:
        raise ValueError("video_id 和 account_id 必填")

    # 检查视频是否存在
    result = await session.execute(select(Video).where(Video.id == video_id))
    video = result.scalar_one_or_none()
    if video is None:
        raise ValueError("视频不存在")

    # 检查是否已点赞（利用 likes 表的联合唯一索引，O(1) 查询）
    result = await session.execute(
        select(Like).where(
            Like.video_id == video_id,
            Like.account_id == account_id,
        )
    )
    if result.scalar_one_or_none() is not None:
        raise ValueError("已经点过赞了")

    # ===== 第2步：尝试异步路径（发MQ消息） =====
    mysql_enqueued = False   # likeMQ 是否发送成功
    redis_enqueued = False   # popularityMQ 是否发送成功

    # 尝试发 likeMQ（写 MySQL 的 Worker 路径）
    if like_mq is not None:
        try:
            await like_mq.publish_like(account_id=account_id, video_id=video_id)
            mysql_enqueued = True
            logging.debug(f"likeMQ 发送成功: user={account_id}, video={video_id}")
        except Exception as e:
            logging.warning(f"likeMQ 发送失败，将降级为直接写MySQL: {e}")
            # ← 不抛异常！继续走降级路径

    # 尝试发 popularityMQ（更新 Redis 热榜的 Worker 路径）
    if popularity_mq is not None:
        try:
            await popularity_mq.publish_update(video_id=video_id, delta=1)
            redis_enqueued = True
            logging.debug(f"popularityMQ 发送成功: video={video_id}, delta=+1")
        except Exception as e:
            logging.warning(f"popularityMQ 发送失败，将直接写Redis: {e}")
            # ← 同样不抛异常！继续走降级路径

    # 两条 MQ 都成功 → 最快路径，直接返回！
    if mysql_enqueued and redis_enqueued:
        return

    # ===== 第3步：降级路径 — MySQL 直写 =====
    # 当 likeMQ 发送失败时，不走 Worker，直接写 MySQL 事务
    if not mysql_enqueued:
        try:
            async with session.begin():
                # 再次确认视频存在（防止并发删除）
                result = await session.execute(
                    select(Video.id).where(Video.id == video_id)
                )
                if result.scalar_one_or_none() is None:
                    raise ValueError("视频已被删除")

                # 插入点赞记录
                session.add(Like(video_id=video_id, account_id=account_id))

                # 原子更新：likes_count + 1
                await session.execute(
                    update(Video)
                    .where(Video.id == video_id)
                    .values(likes_count=Video.likes_count + 1)
                )

                # 原子更新：popularity + 1
                await session.execute(
                    update(Video)
                    .where(Video.id == video_id)
                    .values(popularity=Video.popularity + 1)
                )
            # 事务提交成功
            logging.info(f"MySQL降级写入成功: user={account_id} liked video={video_id}")
        except Exception as e:
            # 1062 是 MySQL 的重复键错误码 → 说明已经点过赞了
            if hasattr(e, 'orig') and getattr(e.orig, 'args', [None])[0] == 1062:
                raise ValueError("已经点过赞了")
            raise

    # ===== 第4步：降级路径 — Redis 直写 =====
    # 当 popularityMQ 发送失败时，直接操作 Redis ZSET

            logging.error(f"Redis降级写入也失败了: {e}")
```

```python
"""
评论 Service — 异步优先 + 降级模式 + @mention 提取通知。
对应 Go 的 internal/video/comment_service.go
"""

import re
import logging
from sqlalchemy import select, update

from app.models.notification import Notification


# 提取 @username 的正则（字母数字下划线，Go项目用的是 \w+）
MENTION_PATTERN = re.compile(r'@(\w+)')


async def publish_comment(
    session: AsyncSession,
    video_id: int,
    author_id: int,
    username: str,
    content: str,
    comment_mq,
    popularity_mq,
    redis_client,
) -> Comment:
    """
    发布评论 — 三件事：
      1. 写评论（异步MQ优先 + MySQL降级）
      2. 更新热度（异步MQ优先 + Redis降级）
      3. 提取 @username → 创建通知记录

    流程类似点赞，多了 @mention 步骤。
    """
    content = content.strip()
    if not content:
        raise ValueError("评论内容不能为空")
    if video_id <= 0 or author_id <= 0:
        raise ValueError("参数不完整")

    # 校验视频存在
    result = await session.execute(select(Video).where(Video.id == video_id))
    if result.scalar_one_or_none() is None:
        raise ValueError("视频不存在")

    # ===== 异步优先路径 =====
    mysql_enqueued = False
    redis_enqueued = False

    if comment_mq is not None:
        try:
            await comment_mq.publish(
                username=username,
                video_id=video_id,
                author_id=author_id,
                content=content,
            )
            mysql_enqueued = True
        except Exception as e:
            logging.warning(f"commentMQ 发送失败，降级直写MySQL: {e}")

    if popularity_mq is not None:
        try:
            await popularity_mq.publish_update(video_id=video_id, delta=1)
            redis_enqueued = True
        except Exception as e:
            logging.warning(f"popularityMQ 发送失败: {e}")

    # ===== 降级路径 — MySQL 直写 =====
    comment = None
    if not mysql_enqueued:
        async with session.begin():
            result = await session.execute(
                select(Video.id).where(Video.id == video_id)
            )
            if result.scalar_one_or_none() is None:
                raise ValueError("视频已被删除")

            comment = Comment(
                username=username,
                video_id=video_id,
                author_id=author_id,
                content=content,
            )
            session.add(comment)
            await session.flush()  # 获取 comment.id

            await session.execute(
                update(Video)
                .where(Video.id == video_id)
                .values(popularity=Video.popularity + 1)
            )

    # ===== @mention 通知处理（无论走哪条路径都要执行） =====
    await _notify_mentions(session, video_id, author_id, username, content)

        try:
            await redis_client.delete(f"v1:video:detail:id={video_id}")

            pass

    return comment


async def _notify_mentions(
    session: AsyncSession,
    video_id: int,
    author_id: int,
    author_username: str,
    content: str,
) -> None:
    """
    扫描评论中的 @username，对被提及的用户创建通知。
    跳过：自己@自己、重复@（同一条评论中对同一个用户只通知一次）。
    """
    mentions = MENTION_PATTERN.findall(content)
    if not mentions:
        return

    # 去重 + 排除自己
    seen: set[str] = set()
    for mentioned_username in mentions:
        if mentioned_username in seen or mentioned_username == author_username:
            continue
        seen.add(mentioned_username)

        # 查找被提及用户的 ID
        from app.models.account import Account
        result = await session.execute(
            select(Account.id).where(Account.username == mentioned_username)
        )
        mentioned_id = result.scalar_one_or_none()
        if mentioned_id is None:
            continue  # 用户名不存在，跳过

        # 创建通知
        session.add(Notification(
            recipient_id=mentioned_id,
            sender_id=author_id,
            type="mention",
            target_id=video_id,
            content=f"{author_username} 在评论中提到了你",
        ))
```

### D. 面试追问预判

#### Q1: "异步优先 + 降级兜底 这个模式有什么缺点？"

"两个缺点：
1. **代码复杂度翻倍**：每个写操作都要维护两套路径（MQ + MySQL），单元测试也要覆盖两种场景。但这是值得的——用代码复杂度换系统可靠性。
2. **最终一致性**：MQ 路径下，点赞后立刻刷新 Feed 可能看到点赞数还是旧的（Worker 还没消费）。对于短视频场景是可接受的——用户不太可能在 0.5 秒内反复刷新。如果需要强一致性（如支付），就不能用这个模式。"

#### Q2: "如果 Worker 消费消息时失败了怎么办？"

"RabbitMQ 配置了死信队列（DLX）+ 最大重试 3 次。消费失败时 Nack 让 MQ 重新投递；3 次都失败后消息进入 DLX 死信队列，不会被丢弃。运维可以查看死信队列排查问题（比如外键约束冲突、数据库连接断开），修复后手动重新投递。"

#### Q3: "为什么点赞不用 Redis 做计数器而是直接更新 MySQL？"

"Redis 做计数器的风险：Redis 重启/宕机会丢失计数。虽然可以用 AOF 持久化，但这里 likes_count 是视频展示的核心数据，不允许丢失。当前方案的取舍：每次点赞多一次 MySQL UPDATE，换取计数的绝对可靠。只有热度分（popularity）对丢失容忍度高，用 Redis ZSET 存储。"

#### Q4: "视频发布时为什么不直接把视频文件信息写入消息队列？"

"视频文件本身（200MB）不走 MQ，MQ 只传视频 ID 和元数据（create_time 等）。文件通过 HTTP multipart/form-data 上传到 API Server，存储到本地磁盘后，play_url 作为视频元数据的一部分写入 MySQL。如果要做 CDN 分发，Worker 消费 outbox 消息时可以触发转码/上传 CDN 的任务。"

> 面试场景：面试官问"Feed 流怎么实现？下拉刷新的下一页怎么取？深分页有什么问题？怎么解决？"

### A. 设计决策

#### A.1 五种 Feed 类型一览

| 类型 | 接口 | 排序字段 | 分页方式 | 是否需要登录 |
|------|------|----------|----------|-------------|
| 最新视频 | `/feed/listLatest` | create_time DESC | **游标分页** | 软鉴权 |
| 点赞排行 | `/feed/listLikesCount` | likes_count DESC | **复合游标分页** | 软鉴权 |
| 热度榜 | `/feed/listByPopularity` | popularity DESC | **快照分页** | 软鉴权 |
| 关注流 | `/feed/listByFollowing` | create_time DESC | 游标分页 + 缓存 | 严格鉴权 |
| 标签流 | `/feed/listByTag` | create_time DESC | 无分页（直接查） | 软鉴权 |

**接口为什么不设计成分页参数统一？** 因为每种排序的数据分布不同：
- `create_time`：单调递增，值几乎不重复 → 简单游标分页即可
- `likes_count`：离散值，大量重复（很多视频都是 0 赞）→ 需要复合游标
- `popularity`：实时变化，翻页时值可能变了 → 需要快照冻结

#### A.2 为什么不用 OFFSET？

这是面试中的高频考点：

```sql
-- OFFSET 分页（不好的做法）
SELECT * FROM videos ORDER BY create_time DESC LIMIT 20 OFFSET 10000;
-- 数据库必须扫描并丢弃前 10000 行，才能返回 20 行
-- OFFSET 越大越慢

-- 游标分页（好的做法）
SELECT * FROM videos WHERE create_time < '2026-01-01 12:00:00'
ORDER BY create_time DESC LIMIT 20;
-- 利用索引直接定位到起始位置，只扫描需要的 20 行
-- 无论"第几页"性能都一样
```

> **MySQL 知识点**：InnoDB 的 B+ 树索引支持范围扫描。`WHERE create_time < X ORDER BY create_time DESC` 可以利用 `idx_create_time` 索引，从 X 的位置开始向前扫描，不需要回表跳页。而 `OFFSET 10000` 虽然也能用索引，但必须沿索引链表走 10000 步才能开始取数据——这个过程纯粹在浪费 I/O。

**那游标分页有什么缺点？**
- 不能跳页（只能"上一页"和"下一页"）
- 但对于 Feed 流（无限下拉）来说，这个缺点无所谓——用户本来就不跳页

#### A.3 为什么点赞排行需要"复合游标"？

单字段游标翻车场景：当排序字段存在重复值时。

```
假设 likes_count 排序，当前页最后一条是：
  {id: 30, likes_count: 5}

下一页 SQL: WHERE likes_count < 5 ORDER BY likes_count DESC LIMIT 10
问题：id=25 的点赞数也是 5，但它排在 id=30 前面（因为 ORDER BY likes_count DESC
     在相同 likes_count 下的顺序是不确定的），可能被重复返回或跳过的视频。

解决方案——复合游标：
  WHERE (likes_count < 5) OR (likes_count = 5 AND id < 30)
  ORDER BY likes_count DESC, id DESC LIMIT 10
  ↑ 第二排序键 id 保证了相同 likes_count 下的稳定顺序
```

### B. 设计思想（Go 参考实现提炼）

Go 项目在 `feed/service.go` 中实现了 400+ 行核心 Feed 逻辑：

- **三级缓存获取视频实体**：`GetVideoByIDs()` 方法用 L1(go-cache) → L2(Redis) → L3(MySQL) 三级架构，是独立的缓存模块
- **冷热分离**：`ListLatest()` 基于 Redis ZSET `feed:global_timeline` 的水位判断走热路径（ZSET）还是冷路径（MySQL）
- **热度榜快照**：`ListByPopularity()` 用 `ZUNIONSTORE` 合并 60 个分钟窗口生成快照，快照 TTL 2 分钟留给翻页

#### C.1 最新视频 Feed：游标分页 + 冷热分离（⭐ 核心算法）

```python
    redis_client,
    singleflight,                     # 防止并发击穿的 singleflight 实例

    like_service,                     # 查点赞状态用
) -> dict:

      {
        "video_list": [...],   # FeedVideoItem 列表（含 is_liked 状态）
        "next_time": 1715000000000,  # 下一页游标（毫秒时间戳）
        "has_more": true             # 是否还有更多数据
      }
    """

    timeline_key = "v1:feed:global_timeline"  # Redis ZSET key

    # ===== 第1步：检查 ZSET 是否为空 =====
    # ZRangeWithScores 取最老的一条（索引0，条数1）来看水位
    if redis_client is not None:
        tail = await redis_client.zrange_withscores(timeline_key, 0, 0)

        # ===== 分支A：ZSET 为空 → 重建 =====
        if not tail:
            # ★ 全局锁：用 singleflight 保证只有一个请求去重建 ZSET
            #    防止 1000 个并发请求各自去 MySQL 捞数据
            sf_key = "sf:fallback:global_timeline_rebuild"
            result = await singleflight.do(sf_key, lambda: _rebuild_timeline(
                session, redis_client, timeline_key
            ))

            if result == "EMPTY":
                # 数据库也没有视频 → 返回空列表
                return {"video_list": [], "next_time": 0, "has_more": False}

            # 重建完成后，递归调用自己走热路径
            return await list_latest(
                session, limit, latest_before, viewer_id,
                redis_client, singleflight, cache_service, like_service,
            )

        # ===== 分支B：ZSET 不为空 → 判断冷热 =====
        # ZSET 中最老数据的时间戳 = 冷热分界线（watermark）
        watermark = int(tail[0][1])  # tail[0][1] 是 score（即 create_time 的毫秒时间戳）

        # 请求游标的时间戳（第一页 = 当前时间）
        req_time = int(latest_before.timestamp() * 1000) if latest_before else None

        # ===== 判断走热路径还是冷路径 =====
        if req_time is not None and req_time <= watermark:
            # ---- 冷数据路径：直接查 MySQL ----
            # 原因：游标已经翻到了 ZSET 覆盖范围之外的历史数据
            # 不回写 ZSET（避免冷数据污染热点时间线）
            return await _cold_path_latest(session, limit, latest_before, viewer_id, like_service)

        else:
            # ---- 热数据路径：查 Redis ZSET ----
            return await _hot_path_latest(
                session, limit, req_time, latest_before, viewer_id,
                redis_client, timeline_key, cache_service, like_service,
            )

    # Redis 不可用 → 直接查 MySQL（无冷热分离的简化路径）
    return await _fallback_latest(session, limit, latest_before, viewer_id, like_service)


# ================================
# 辅助函数：重建全局时间线 ZSET
# ================================
async def _rebuild_timeline(session, redis_client, timeline_key: str) -> str:
    """
    从 MySQL 捞最新 1000 条视频 → 重建 feed:global_timeline ZSET。
    member = video_id (string), score = create_time 的毫秒时间戳。
    """
    result = await session.execute(
        select(Video.id, Video.create_time)
        .order_by(Video.create_time.desc())
        .limit(1000)
    )
    rows = result.all()
    if not rows:
        return "EMPTY"

    # 批量 ZADD：一次性写入所有 member+score
    mapping = {
        str(row.id): row.create_time.timestamp() * 1000
        for row in rows
    }
    await redis_client.zadd(timeline_key, mapping)
    logging.info(f"Timeline ZSET 重建完成: {len(mapping)} 条视频")
    return "OK"


# ================================
# 热数据路径：Redis ZSET → 三级缓存
# ================================
async def _hot_path_latest(
    session, limit, req_time, latest_before, viewer_id,
    redis_client, timeline_key, cache_service, like_service,
) -> dict:
    """
    热路径：从 Redis ZSET 取视频 ID → 三级缓存获取完整实体。

    ZSET 的 score 是 create_time 毫秒时间戳，member 是 video_id 字符串。
    用 ZREVRANGEBYSCORE 按 score 倒序取 ID 列表。
    """
    # 确定 score 范围：从上一页游标开始（不包含），到 -inf
    if req_time is not None:
        max_score = req_time - 1  # 减 1ms，不重复返回上一页的最后一条
    else:
        max_score = "+inf"

    # ZREVRANGEBYSCORE：按 score 倒序取 limit 条
    ids_str = await redis_client.zrevrangebyscore(
        timeline_key,
        max_score=str(max_score),
        min_score="-inf",
        offset=0,
        count=limit,
    )

    # 转换 video_id 类型
    video_ids = [int(x) for x in ids_str] if ids_str else []

    # 三级缓存获取视频实体（下一层详讲，这里先简化为直接查库）
    if video_ids:
        videos = await cache_service.get_videos_by_ids(video_ids)
    else:
        videos = []

    # ===== 冷热边界缝合 =====
    # 如果 ZSET 返回的视频不够 limit 条，说明击穿了冷热边界
    if len(videos) < limit:
        remain = limit - len(videos)
        # 确定冷数据的游标：取已获得数据的最后一条时间
        cold_cursor = videos[-1].create_time if videos else latest_before

        # 用 singleflight 防并发（冷边界可能是多人同时翻到这里）
        sf_key = f"sf:stitch:listLatest:{remain}:{int(cold_cursor.timestamp()*1000)}"
        cold_videos = await singleflight.do(sf_key, lambda: _query_cold_latest(
            session, remain, cold_cursor
        ))
        videos.extend(cold_videos if cold_videos else [])

    # 组装 FeedVideoItem（含 is_liked 状态）
    feed_items = await _build_feed_items(videos, viewer_id, like_service)

    has_more = len(videos) == limit
    next_time = int(videos[-1].create_time.timestamp() * 1000) if videos else 0

    return {"video_list": feed_items, "next_time": next_time, "has_more": has_more}


# ================================
# 冷数据路径：直接查 MySQL
# ================================
async def _cold_path_latest(session, limit, latest_before, viewer_id, like_service) -> dict:
    """冷路径：直接查 MySQL，不回写 ZSET"""
    videos = await _query_cold_latest(session, limit, latest_before)
    feed_items = await _build_feed_items(videos, viewer_id, like_service)
    has_more = len(videos) == limit
    next_time = int(videos[-1].create_time.timestamp() * 1000) if videos else 0
    return {"video_list": feed_items, "next_time": next_time, "has_more": has_more}


async def _query_cold_latest(session, limit: int, before: datetime | None):
    """查最新视频的 SQL（游标分页）"""
    stmt = select(Video).order_by(Video.create_time.desc())
    if before:
        stmt = stmt.where(Video.create_time < before)
    stmt = stmt.limit(limit)
    result = await session.execute(stmt)
    return list(result.scalars().all())
```

```python
# app/services/feed.py — ListLikesCount

async def list_likes_count(
    session: AsyncSession,
    limit: int,
    cursor: tuple[int, int] | None,  # (likes_count_before, id_before) 复合游标
    viewer_id: int,
    like_service,
) -> dict:
    """
    点赞排行 Feed — 复合游标分页。

    游标结构：(likes_count_before, id_before)
    SQL 条件：
      WHERE (likes_count < ?) OR (likes_count = ? AND id < ?)
      ORDER BY likes_count DESC, id DESC

    为什么需要 id 作为第二排序键？
      → 点赞数相同时，id 保证排序稳定，不会出现跳页/重复

    返回：
      {
        "video_list": [...],
        "next_likes_count_before": 5,
        "next_id_before": 30,
        "has_more": true
      }
    """
    # 构建查询
    stmt = select(Video).order_by(
        Video.likes_count.desc(),  # 主排序键
        Video.id.desc(),            # 第二排序键（保证稳定性）
    )

    if cursor is not None:
        likes_before, id_before = cursor
        # ★ 复合游标的 WHERE 条件：
        #    (likes_count < X)   ← 点赞数比游标小的
        #    OR (likes_count = X AND id < Y)  ← 点赞数相同但 id 更小的
        from sqlalchemy import or_, and_
        stmt = stmt.where(
            or_(
                Video.likes_count < likes_before,
                and_(Video.likes_count == likes_before, Video.id < id_before),
            )
        )

    stmt = stmt.limit(limit)
    result = await session.execute(stmt)
    videos = list(result.scalars().all())

    # 组装响应
    feed_items = await _build_feed_items(videos, viewer_id, like_service)
    has_more = len(videos) == limit

    response = {"video_list": feed_items, "has_more": has_more}

    if videos:
        last = videos[-1]
        response["next_likes_count_before"] = last.likes_count
        response["next_id_before"] = last.id

    return response
```

#### C.3 热度榜 Feed：快照分页

```python
# app/services/feed.py — ListByPopularity

async def list_by_popularity(
    session: AsyncSession,
    limit: int,
    as_of: int,       # 快照时刻的分钟时间戳（第一页传0，后续传服务端返回的值）
    offset: int,       # 当前偏移量（第一页传0）
    viewer_id: int,
    redis_client,
    like_service,
) -> dict:
    """
    热度榜 Feed — 快照分页（解决翻页抖动问题）。

    为什么需要快照？
      热榜数据实时变化。如果不冻结时间点，翻第2页时前20名已经变了，
      第1页的最后一条可能被挤出排名 → 第2页出现重复视频。

    快照机制：
      1. 确定 as_of 分钟（第一页用当前分钟）
      2. ZUNIONSTORE 合并最近 60 个分钟窗口 → 生成快照
      3. 快照 TTL 2分钟（留翻页时间）
      4. ZREVRANGE 按 offset 分页读取快照

    Redis Key 体系：
      写入侧: hot:video:1m:{YYYYMMDDHHMM}     ← 每分钟一个ZSET（滑动窗口）
      查询侧: hot:video:merge:1m:{YYYYMMDDHHMM}  ← 合并60个窗口的快照
    """
    if redis_client is None:
        # Redis 不可用 → 降级为 MySQL 三字段游标分页
        return await _fallback_popularity(session, limit, viewer_id, like_service)

    from datetime import datetime, timedelta, timezone

    # 1. 确定快照时刻
    if as_of > 0:
        as_of_dt = datetime.fromtimestamp(as_of, tz=timezone.utc).replace(second=0, microsecond=0)
    else:
        as_of_dt = datetime.now(timezone.utc).replace(second=0, microsecond=0)

    # 2. 构建 60 个分钟窗口的 Key 列表
    window_keys = []
    for i in range(60):
        minute = as_of_dt - timedelta(minutes=i)
        window_keys.append(f"v1:hot:video:1m:{minute.strftime('%Y%m%d%H%M')}")

    # 3. ZUNIONSTORE 合并 → 生成快照
    dest_key = f"v1:hot:video:merge:1m:{as_of_dt.strftime('%Y%m%d%H%M')}"

    # 检查快照是否已存在（同一 as_of 的翻页复用同一个快照）
    exists = await redis_client.exists(dest_key)
    if not exists:
        # SUM 聚合：每个 video_id 在 60 个窗口中的 score 求和
        await redis_client.zunionstore(dest_key, window_keys, aggregate="SUM")
        # TTL 2 分钟：给翻页留时间，过期后下次请求重新生成（拿到最新排名）
        await redis_client.expire(dest_key, 120)

    # 4. ZREVRANGE 按 offset 分页
    members = await redis_client.zrevrange(dest_key, offset, offset + limit - 1)
    if not members:
        return {"video_list": [], "as_of": int(as_of_dt.timestamp()), "next_offset": offset, "has_more": False}

    # 5. 用 video_ids 查 MySQL 获取完整实体
    video_ids = [int(m) for m in members]
    result = await session.execute(
        select(Video).where(Video.id.in_(video_ids))
    )
    videos_map = {v.id: v for v in result.scalars().all()}

    # 保持 ZSET 排序顺序（ZSET 按 score 排，MySQL IN 查询不保证顺序）
    ordered = [videos_map[vid] for vid in video_ids if vid in videos_map]

    feed_items = await _build_feed_items(ordered, viewer_id, like_service)
    has_more = len(members) == limit

    return {
        "video_list": feed_items,
        "as_of": int(as_of_dt.timestamp()),
        "next_offset": offset + len(feed_items),
        "has_more": has_more,
    }


async def _fallback_popularity(session, limit, viewer_id, like_service) -> dict:
    """
    热度榜 MySQL 降级路径 — 三字段复合游标分页。

    ORDER BY popularity DESC, create_time DESC, id DESC
    需要三个游标字段：(latest_popularity, latest_before, latest_id_before)
    """
    stmt = select(Video).order_by(
        Video.popularity.desc(),
        Video.create_time.desc(),
        Video.id.desc(),
    ).limit(limit)
    result = await session.execute(stmt)
    videos = list(result.scalars().all())
    feed_items = await _build_feed_items(videos, viewer_id, like_service)
    return {"video_list": feed_items, "has_more": len(videos) == limit}
```

```python
# app/services/feed.py — _build_feed_items

async def _build_feed_items(
    videos: list[Video],
    viewer_id: int,
    like_service,
) -> list[dict]:
    """
    将 Video 实体列表 → FeedVideoItem 列表。
    核心优化：批量查询点赞状态，避免 N+1 查询。

    N+1 问题示例（不好的做法）：
      for video in videos:
          is_liked = await like_service.is_liked(video.id, viewer_id)  # ← 每个视频一次DB查询

    批量查询（好的做法）：
      video_ids = [v.id for v in videos]
      liked_map = await like_service.batch_get_liked(video_ids, viewer_id)  # ← 一次DB查询
    """
    if not videos:
        return []

    # ★ 批量查询点赞状态（一次 IN 查询，而非 N 次单独查询）
    video_ids = [v.id for v in videos]
    liked_map = await like_service.batch_get_liked(video_ids, viewer_id)
    # liked_map = {video_id_1: True, video_id_2: False, ...}

    feed_items = []
    for v in videos:
        feed_items.append({
            "id": v.id,
            "author": {"id": v.author_id, "username": v.username},
            "title": v.title,
            "description": v.description,
            "play_url": v.play_url,
            "cover_url": v.cover_url,
            "create_time": int(v.create_time.timestamp()),
            "likes_count": v.likes_count,
            "is_liked": liked_map.get(v.id, False),  # 匿名用户全部 False
        })

    return feed_items
```

```python
async def list_by_following(
    session: AsyncSession,
    limit: int,
    latest_before: datetime | None,
    viewer_id: int,           # 必须登录，否则没有关注列表
    redis_client,
    like_service,
) -> dict:
    """
    关注流 — 查出我关注的人发的视频，按时间排序。

    缓存策略：关注流是"热数据"，关注关系不变时结果不常变，用 Redis 缓存。
    Key 格式: feed:listByFollowing:limit={n}:accountID={id}:before={t}

    分布式锁防击穿：和鉴权缓存同样的 SETNX 模式。
    """
    if viewer_id == 0:
        return {"video_list": [], "next_time": 0, "has_more": False}

    # ===== 尝试读缓存 =====
    if redis_client is not None:
        before_key = int(latest_before.timestamp()) if latest_before else 0
        cache_key = f"v1:feed:listByFollowing:limit={limit}:accountID={viewer_id}:before={before_key}"

        cached = await redis_client.get(cache_key)
        if cached is not None:
            import json
            return json.loads(cached)

        # 缓存未命中 → SETNX 抢锁
        lock_key = f"lock:{cache_key}"
        lock_token = await redis_client.setnx(lock_key, "1", ex=500)  # 500ms TTL
        if lock_token:
            # 获得锁 → 查 DB → 写缓存
            resp = await _query_following(session, limit, latest_before, viewer_id, like_service)
            try:
                import json
                await redis_client.set(cache_key, json.dumps(
                    {"video_list": resp[0], "next_time": resp[1], "has_more": resp[2]}
                ), ex=5)  # 缓存 TTL 5秒
            except Exception:
                pass
            return {"video_list": resp[0], "next_time": resp[1], "has_more": resp[2]}
        else:
            # 没抢到锁 → 等待 5 次重试读缓存
            for _ in range(5):
                await asyncio.sleep(0.02)
                cached = await redis_client.get(cache_key)
                if cached is not None:
                    import json
                    return json.loads(cached)

    # ===== 缓存归源：查 DB =====
    videos, has_more, next_time = await _query_following(
        session, limit, latest_before, viewer_id, like_service
    )
    return {"video_list": videos, "next_time": next_time, "has_more": has_more}


async def _query_following(session, limit, before, viewer_id, like_service):
    """查关注者的视频 — 子查询模式"""
    # 子查询：SELECT vlogger_id FROM socials WHERE follower_id = ?
    from app.models.social import Social
    following_subq = (
        select(Social.vlogger_id)
        .where(Social.follower_id == viewer_id)
    )

    stmt = (
        select(Video)
        .where(Video.author_id.in_(following_subq))
        .order_by(Video.create_time.desc())
        .limit(limit)
    )
    if before:
        stmt = stmt.where(Video.create_time < before)

    result = await session.execute(stmt)
    videos = list(result.scalars().all())
    feed_items = await _build_feed_items(videos, viewer_id, like_service)
    has_more = len(videos) == limit
    next_time = int(videos[-1].create_time.timestamp()) if videos else 0
    return feed_items, has_more, next_time
```

### D. 面试追问预判

#### Q1: "游标分页和 OFFSET 分页各自适用于什么场景？"

| 维度 | 游标分页 | OFFSET 分页 |
|------|---------|------------|
| 性能 | 不受数据量影响，O(log n) 定位 + O(limit) 扫描 | 随 OFFSET 增大线性变慢 |
| 跳页 | 不支持 | 支持 |
| 实时数据 | 数据增删时"自动修正"（跳过已删除的） | 数据增删时结果会飘移 |
| 适用场景 | Feed 流、消息列表（无限下拉） | 后台管理、搜索结果（需要跳页） |

#### Q2: "冷热分离的水位（1000条）怎么定的？"

"1000 条是根据预估的活跃用户 × 平均刷视频数来估算的。如果 100 个活跃用户每人刷 50 条视频，1000 条热量可以覆盖一部分用户的完整浏览。水位不是固定的——可以做成配置项，根据实际数据调整：太小则频繁触发冷路径（慢），太大则 ZSET 内存占用过高。生产环境可以通过监控 ZSET 的命中率来调优。"

#### Q3: "如果 ZSET 重建期间有新视频发布怎么办？"

"新发布的视频通过 outbox → timeline MQ → TimelineConsumer 实时 ZADD 到 `feed:global_timeline`。ZSET 重建用的 `singleflight` 只是防止并发重建，不阻止正常写入。重建完成后的 ZSET 包含了重建期间新增的视频（因为重建是从 MySQL 查的，新视频已经写入了 DB）。旧数据可能被多写一次，但 ZADD 是幂等的（相同的 member 会更新 score），不影响正确性。"

#### Q4: "快照的 TTL 设为 2 分钟，如果用户在 2 分钟后翻下一页怎么办？"

"用户翻页时带的 `as_of` 对应快照 Key 已经过期了。这时服务端检测到 dest key 不存在，重新执行 ZUNIONSTORE 生成新快照。新快照的排名可能和 2 分钟前略有不同（热度实时变化），但用户体验上影响不大——翻了几页之后的数据本来就在榜单底部，微小波动不可感知。"

### A. 设计决策

#### A.1 Redis 在这个系统中的五重身份

```
┌─────────────────────────────────────────────────────────────┐
│                     Redis 的五重身份                         │
├─────────────┬────────────────┬──────────────────────────────┤
│ 身份        │ Key 模式        │ 服务的目标                    │
├─────────────┼────────────────┼──────────────────────────────┤
│ 鉴权缓存    │ account:<id>   │ 避免每次请求都查MySQL验Token  │
│ 数据缓存    │ video:entity:<id> │ 视频详情三级缓存的L2层     │
│ Feed缓存    │ feed:listLatest:* │ 匿名Feed流短TTL缓存        │
│ 热榜存储    │ hot:video:1m:* │ 滑动窗口ZSET，排名数据持久化  │
│ 分布式锁    │ lock:*         │ SETNX防缓存击穿              │
│ 限流计数器  │ feedsystem:ratelimit:* │ 滑动窗口计数         │
└─────────────┴────────────────┴──────────────────────────────┘
```

#### A.2 三个缓存问题的本质

面试中一定会被问到这三个概念。它们的根本原因不同，解决方案也不同：

| 问题 | 现象 | 原因 | 解决方案 |
|------|------|------|----------|
| **缓存穿透** | 查一个不存在的 key，每次都穿到 DB | 恶意请求 / 无效 ID | 布隆过滤器 / 缓存空值 |
| **缓存击穿** | 热点 key 过期瞬间，大量请求同时打到 DB | 并发请求同一 key | 分布式锁（SETNX），只让一个人回源 |
| **缓存雪崩** | 大量 key 同时过期，DB 瞬间被打爆 | TTL 设成了一样的值 | TTL 加随机值 |

**一个比喻帮助记忆**：
- 穿透 = 打不存在的靶子（每次都要去仓库确认没有这个靶子）
- 击穿 = 一个热门靶子被撤掉，所有人挤去仓库
- 雪崩 = 整个靶场的靶子同时撤掉，仓库被挤爆

#### A.3 三级缓存的层次设计

```
请求 video_id=42 的实体
    │
    ▼
┌───────────────┐  命中率~70%  延迟~0.01ms
│ L1: 本地内存   │  cachetools.TTLCache / go-cache
│ TTL: 3秒      │  进程内，无网络开销，最快但容量最小
└───────┬───────┘
        │ 未命中
        ▼
┌───────────────┐  命中率~25%  延迟~0.5ms
│ L2: Redis     │  STRING key=video:entity:42, value=JSON
│ TTL: 1小时    │  跨进程共享，中等速度，容量较大
└───────┬───────┘
        │ 未命中
        ▼
┌───────────────┐  命中率~5%   延迟~5ms
│ L3: MySQL     │  SELECT * FROM videos WHERE id=42
│ 永久存储       │  最慢但最可靠，source of truth
└───────────────┘

每一层都在上面一层的"miss"时才被访问。
每一层命中后都会回填上层（L3→异步写L2→写L1）。
```

**为什么 L1 TTL 只有 3 秒？** 本地缓存是进程内的，多个 API Server 实例之间不共享。如果 TTL 太长，实例 A 修改了数据但实例 B 还在用旧缓存。3 秒的短 TTL 把不一致窗口控制在可接受的范围。

---

### B. 设计思想（Go 参考实现提炼）

Go 项目用 `patrickmn/go-cache` 做 L1（3s TTL），Redis 做 L2（1h TTL），MySQL 做 L3。每层都用了 `singleflight`（`golang.org/x/sync/singleflight`）防击穿——同一个 key 的并发请求合并为一次实际查询。

#### C.1 L1 本地缓存：cachetools.TTLCache

```python
# app/core/local_cache.py
"""
L1 本地内存缓存 — 基于 cachetools.TTLCache。

特点：
  - 进程内缓存，无网络开销，延迟最低（微秒级）
  - TTL 到期自动淘汰
  - 有最大容量限制，防止 OOM
  - 多个 API 实例之间不共享（这是 L2 Redis 存在的理由）

对应 Go 的 github.com/patrickmn/go-cache
"""

from cachetools import TTLCache
import threading

# maxsize=10000：最多存 10000 个视频实体
# ttl=3：每个条目 3 秒后自动过期
# 10000 个 Video JSON 对象 ≈ 内存占用约 20-50MB
local_cache = TTLCache(maxsize=10000, ttl=3)

# TTLCache 本身不是线程安全的（asyncio 场景也能遇到并发问题），
# 加一把轻量锁保护。在 Python 中 asyncio.Lock 是协程级锁，
# threading.Lock 是线程级锁。这里用 threading.Lock 兼容同步和异步调用。
_cache_lock = threading.Lock()


def l1_get(key: str) -> dict | None:
    """从 L1 读取，miss 返回 None"""
    with _cache_lock:
        return local_cache.get(key)


def l1_set(key: str, value: dict):
    """写入 L1（自动 TTL 3秒）"""
    with _cache_lock:
        local_cache[key] = value


def l1_delete(key: str):
    """主动失效"""
    with _cache_lock:
        local_cache.pop(key, None)
```

#### C.2 L2 Redis 缓存封装（详细操作）

```python
# app/core/redis.py（缓存部分摘录）
"""
Redis 客户端封装 — 提供 STRING / ZSET / Lock 等操作。

对应 Go 的 internal/middleware/redis/ 下的三个文件：
  redis.go  — 连接 + Lock + IncrementWithExpire
  cache.go  — GetBytes / SetBytes / Del / MGet
  zset.go   — ZincrBy / ZAdd / ZUnionStore / ZRevRange 等
"""

import json
import logging
import secrets
from typing import Any
from redis.asyncio import Redis


class RedisClient:
    """Redis 客户端封装，所有方法都是 async"""

    def __init__(self, redis: Redis, key_prefix: str = "v1:"):
        self._redis = redis
        self._key_prefix = key_prefix  # 所有 key 加前缀，避免和其他系统冲突

    # ==================== 基础 Key 格式化 ====================
    def k(self, fmt: str, *args) -> str:
        """生成带前缀的 Redis key"""
        return self._key_prefix + fmt % args

    # ==================== STRING 操作（缓存用） ====================

    async def get_json(self, key: str) -> dict | None:
        """读取 JSON 缓存，miss 返回 None"""
        raw = await self._redis.get(key)
        if raw is None:
            return None
        try:
            return json.loads(raw)
        except json.JSONDecodeError:
            return None

    async def set_json(self, key: str, value: Any, ttl: int = 3600):
        """写入 JSON 缓存，默认 TTL 1 小时"""
        await self._redis.set(key, json.dumps(value, ensure_ascii=False), ex=ttl)

    async def mget_json(self, keys: list[str]) -> list[dict | None]:
        """批量读取 JSON 缓存（对应 Go 的 MGet）"""
        if not keys:
            return []
        raw_list = await self._redis.mget(keys)
        result = []
        for raw in raw_list:
            if raw is None:
                result.append(None)
            else:
                try:
                    result.append(json.loads(raw))
                except json.JSONDecodeError:
                    result.append(None)
        return result

    async def delete(self, *keys: str):
        """删除缓存"""
        if keys:
            await self._redis.delete(*keys)

    async def exists(self, key: str) -> bool:
        """检查 key 是否存在"""
        return await self._redis.exists(key) > 0

    # ==================== ZSET 操作（热榜用） ====================

    async def zincrby(self, key: str, member: str, increment: float):
        """ZINCRBY：增加成员的分数（不存在则自动创建）"""
        await self._redis.zincrby(key, increment, member)

    async def zadd(self, key: str, mapping: dict[str, float]):
        """ZADD：批量添加成员，mapping = {member: score, ...}"""
        if mapping:
            await self._redis.zadd(key, mapping)

    async def zrevrange(self, key: str, start: int, stop: int) -> list[str]:
        """ZREVRANGE：按 score 倒序取成员（用于分页读热榜）"""
        return await self._redis.zrevrange(key, start, stop)

    async def zrevrangebyscore(
        self, key: str, max_score: str, min_score: str, offset: int, count: int
    ) -> list[str]:
        """ZREVRANGEBYSCORE：按 score 范围倒序取成员（用于 Feed 热路径）"""
        return await self._redis.zrevrangebyscore(
            key, max=max_score, min=min_score, start=offset, num=count
        )

    async def zrange_withscores(self, key: str, start: int, stop: int) -> list[tuple[str, float]]:
        """ZRANGE WITHSCORES：取成员和分数（用于判断 ZSET 水位）"""
        return await self._redis.zrange(key, start, stop, withscores=True)

    async def zunionstore(self, dest: str, keys: list[str], aggregate: str = "SUM"):
        """ZUNIONSTORE：合并多个 ZSET 到目标 key（用于热榜快照生成）"""
        if keys:
            await self._redis.zunionstore(dest, keys, aggregate=aggregate)

    async def expire(self, key: str, ttl: int):
        """设置 TTL（秒）"""
        await self._redis.expire(key, ttl)

    async def zremrangebyrank(self, key: str, start: int, stop: int):
        """ZREMRANGEBYRANK：按排名删除成员（用于裁剪 ZSET 大小）"""
        await self._redis.zremrangebyrank(key, start, stop)

    # ==================== 分布式锁（防缓存击穿） ====================

    async def lock_acquire(self, key: str, ttl: int = 2) -> str | None:
        """
        SETNX 分布式锁。
        返回 token（解锁凭证）表示抢到锁，返回 None 表示没抢到。

        注意：TTL 设置很关键——
          - 太短：操作还没做完锁就过期，其他请求也来查 DB
          - 太长：持有锁的进程崩溃，锁永远不释放（所以必须设 TTL）
        """
        token = secrets.token_hex(16)  # 随机令牌，只有持有者能解锁
        ok = await self._redis.set(key, token, nx=True, ex=ttl)
        return token if ok else None

    async def lock_release(self, key: str, token: str):
        """
        释放分布式锁 — 用 Lua 脚本保证原子性。
        只有持有 token 的人才能释放，防止误删别人的锁。

        Lua 脚本的作用：把"检查是否自己的锁"和"删除"合并为原子操作。
        如果分成两步（先 GET 后 DEL），中间可能被其他请求插入，导致误删。
        """
        script = """
        if redis.call("GET", KEYS[1]) == ARGV[1] then
            return redis.call("DEL", KEYS[1])
        else
            return 0
        end
        """
        await self._redis.eval(script, 1, key, token)

    # ==================== 滑动窗口限流 ====================

    async def increment_with_expire(self, key: str, ttl_ms: int) -> int:
        """
        INCR + 首次设置 TTL — 用 Lua 脚本保证原子性。
        每次请求 INCR，只有第一次请求时设置过期时间。
        这样窗口从第一次请求开始计时，不是固定时间窗口。

        对应 Go 的 IncrementWithExpire。
        """
        script = """
        local count = redis.call("INCR", KEYS[1])
        if count == 1 then
            redis.call("PEXPIRE", KEYS[1], ARGV[1])
        end
        return count
        """
        count = await self._redis.eval(script, 1, key, ttl_ms)
        return int(count)
```

> **OS/Redis 知识点 — Lua 脚本为什么是原子的？** Redis 是单线程执行命令的（主线程只做命令执行，I/O 多路复用在其他线程）。当一个 Lua 脚本开始执行后，Redis 不会插入任何其他命令，直到脚本执行完毕。这保证了 `lock_release` 中"检查"和"删除"之间不会有其他命令插入——类似于操作系统的**临界区**概念，Redis 主线程就是那个保证互斥的机制。

#### C.3 三级缓存：Video 实体获取的完整实现 ⭐

```python
# app/core/cache_service.py
"""
Video 视频实体的三级缓存获取。
对应 Go 的 internal/feed/service.go:GetVideoByIDs()

三级结构：
  L1: 本地内存 (cachetools.TTLCache, TTL 3s)
  L2: Redis STRING key=video:entity:{id}, value=JSON (TTL 1h)
  L3: MySQL SELECT * FROM videos WHERE id IN (...)

每层命中后回填上层（L3→异步写L2→写L1）。
每层用 singleflight 防止同一 key 的并发请求重复回源。
"""

import asyncio
import json
import logging
from app.core.local_cache import l1_get, l1_set


class VideoCacheService:
    """视频实体三级缓存服务"""

    def __init__(self, redis_client, db_session_factory):
        self.redis = redis_client
        self.db_factory = db_session_factory
        self._inflight: dict[str, asyncio.Event] = {}  # singleflight 飞行中记录
        self._results: dict[str, any] = {}              # singleflight 结果存储

    async def get_videos_by_ids(self, video_ids: list[int]) -> list:
        """
        三级缓存获取视频实体列表。
        输入：video_id 列表（可能来自 ZSET、MySQL 查询等）
        输出：Video 对象列表（保持输入顺序）

        如果输入 ids = [1, 2, 3]：
          1. 先批量查 L1（本地内存）
          2. L1 miss 的批量查 L2（Redis MGET）
          3. L2 miss 的逐个用 singleflight 查 L3（MySQL）
        """
        if not video_ids:
            return []

        result_map: dict[int, any] = {}  # {video_id: Video对象}

        # ===== L1: 批量查本地内存 =====
        missed_l1: list[int] = []
        for vid in video_ids:
            cache_key = f"video:entity:{vid}"
            v = l1_get(cache_key)
            if v is not None:
                result_map[vid] = v
            else:
                missed_l1.append(vid)

        if not missed_l1:
            # 全部命中 L1！最快路径
            return [result_map[vid] for vid in video_ids if vid in result_map]

        # ===== L2: 批量查 Redis =====
        missed_l2: list[int] = []
        if self.redis is not None and missed_l1:
            redis_keys = [f"v1:video:entity:{vid}" for vid in missed_l1]
            try:
                redis_results = await self.redis.mget_json(redis_keys)
            except Exception as e:
                logging.warning(f"L2 Redis MGET 失败: {e}")
                # Redis 挂了 → 全部降级到 L3
                missed_l2 = missed_l1
                redis_results = [None] * len(missed_l1)

            if redis_results:
                for i, data in enumerate(redis_results):
                    vid = missed_l1[i]
                    if data is not None:
                        result_map[vid] = data
                        # ★ 回写 L1：Redis 命中的数据补充到本地缓存
                        l1_set(f"video:entity:{vid}", data)
                    else:
                        missed_l2.append(vid)
        else:
            missed_l2 = missed_l1

        if not missed_l2:
            return [result_map[vid] for vid in video_ids if vid in result_map]

        # ===== L3: MySQL 回源（每个 ID 用 singleflight 防击穿） =====
        tasks = []
        for vid in missed_l2:
            # 每个 video_id 作为一个独立的 singleflight key
            # 这样并发请求 video_id=42 的会被合并，但 video_id=43 的不受影响
            sf_key = f"sf:entity:{vid}"
            tasks.append(self._singleflight_l3(sf_key, vid))

        l3_results = await asyncio.gather(*tasks)

        for vid, data in zip(missed_l2, l3_results):
            if data is not None:
                result_map[vid] = data

        # 按输入顺序返回
        return [result_map[vid] for vid in video_ids if vid in result_map]

    async def _singleflight_l3(self, sf_key: str, video_id: int):
        """
        singleflight 防击穿：同一个 key 的并发请求只执行一次 L3 回源。

        工作原理：
          请求A 第一个到达 → 没有 inflight 记录 → 执行 fn，结果存入 _results
          请求B 紧接着到达 → 发现 _inflight[sf_key] 存在 → await Event.wait()
             → A 执行完毕 → Event.set() → B 被唤醒 → 从 _results 拿结果
        """
        # 如果已经有其他协程在查同一个 key，等待它的结果
        if sf_key in self._inflight:
            await self._inflight[sf_key].wait()
            return self._results.get(sf_key)

        # 我是第一个：创建 Event 标记"飞行中"
        self._inflight[sf_key] = asyncio.Event()

        try:
            # 真正执行 L3 回源
            async with self.db_factory() as session:
                from sqlalchemy import select
                from app.models.video import Video
                result = await session.execute(
                    select(Video).where(Video.id == video_id)
                )
                video = result.scalar_one_or_none()

            if video is None:
                self._results[sf_key] = None
                return None

            # 将 ORM 对象转为 dict（用于缓存序列化）
            data = {
                "id": video.id,
                "author_id": video.author_id,
                "username": video.username,
                "title": video.title,
                "description": video.description,
                "play_url": video.play_url,
                "cover_url": video.cover_url,
                "create_time": video.create_time.isoformat(),
                "likes_count": video.likes_count,
                "popularity": video.popularity,
            }

            # ★ 异步回写 L2（不等待，fire-and-forget）
            if self.redis is not None:
                asyncio.create_task(
                    self.redis.set_json(f"v1:video:entity:{video_id}", data, ttl=3600)
                )

            # ★ 同步写 L1（立即生效）
            l1_set(f"video:entity:{video_id}", data)

            self._results[sf_key] = data
            return data

        finally:
            # 唤醒所有等待这个 key 的协程
            self._inflight[sf_key].set()
            # 不立即清理 _results，给等待者读取的时间
```

#### C.4 缓存更新策略：Cache-Aside 模式 + 主动失效

```python
# app/core/cache_service.py（续）

class CacheInvalidation:
    """
    缓存失效策略 — Cache-Aside 模式的核心写部分。

    核心原则：先写数据库，再删缓存（不是先删缓存再写数据库）。

    为什么不能先删缓存再写数据库？（并发写读导致永久脏数据）
      时刻1. 请求A（写）：DELETE cache(key)       ← 缓存被删了
      时刻2. 请求B（读）：GET cache(key) → miss → SELECT from DB → 拿到【旧数据】
             （此时A还没写完，所以B读到的是旧值）
      时刻3. 请求B（读）：SET cache(key, 旧数据)   ← 把旧数据写回了缓存！
      时刻4. 请求A（写）：UPDATE DB SET ...        ← 新数据终于写入DB
      → 最终状态：DB里是新数据，缓存里是B写进去的旧数据 → 永久不一致！
      → 只有等缓存自然过期（可能1小时后），不一致才会消除。

    正确顺序（先写DB后删缓存）：
      时刻1. 请求A（写）：UPDATE DB SET ...        ← 新数据写入DB
      时刻2. 请求B（读）：GET cache(key) → 此时有两种情况：
              ├─ 缓存还没过期 → HIT（拿到旧数据）→ A还没来得及删缓存
              │   时刻3. 请求A（写）：DELETE cache(key) → 缓存被清掉
              │   → 短暂不一致窗口仅存在于时刻2~3之间（微秒级），B的这次读取返回了旧值
              │   → 下一次读请求会 miss，从DB拿到新数据
              └─ 缓存已过期 → miss → SELECT from DB → 拿到【新数据】（因为A已经写完）
                  → 回写缓存时写的是新数据 ✓
      → 不一致窗口极小（DB写完到缓存删除之间的几十微秒），且没有永久脏数据。

    彻底消除不一致需要：
      - 双删策略（写前删 → 写DB → 等几百ms → 再删一次，覆盖并发读的时序窗口）
      - 或订阅 MySQL binlog（Canal/Debezium）异步更新缓存，解耦写DB和失效缓存的原子性
    """

    @staticmethod
    async def on_video_updated(redis_client, video_id: int):
        """视频更新/删除时：失效所有相关缓存"""
        if redis_client is None:
            return

        # 失效视频详情缓存
        await redis_client.delete(f"v1:video:detail:id={video_id}")

        # 失效视频实体缓存（L2）
        await redis_client.delete(f"v1:video:entity:{video_id}")

        # 失效本地缓存（L1）
        l1_delete(f"video:entity:{video_id}")
        l1_delete(f"video:detail:id={video_id}")

    @staticmethod
    async def on_like_changed(redis_client, video_id: int):
        """点赞/取消点赞时：失效视频相关缓存（计数变了）"""
        if redis_client is None:
            return

        await redis_client.delete(
            f"v1:video:detail:id={video_id}",
        )

        # 失效所有 Feed 缓存（点赞数变了，Feed 排序可能受影响）
        # 注意：这是一个激进策略——一个视频的点赞变化 ≈ 所有 Feed 缓存都可能失效
        # 更精细的做法是只失效包含这个视频的缓存页，但实现复杂度高
        # 短 TTL（5s）的 Feed 缓存让这个问题影响很小
```

#### C.5 热榜写入侧：滑动窗口 ZINCRBY

```python
# app/core/popularity.py
"""
热度更新 — 滑动窗口写入侧。

视频热度在写入侧以分钟为单位分桶：
  ZINCRBY hot:video:1m:{YYYYMMDDHHMM} <video_id> <delta>

为什么按分钟分桶？
  1. 分散写入竞争：如果不分桶，所有热度更新都写同一个 ZSET key，
     高并发下 Redis 单线程会成为瓶颈。
  2. 方便滑动窗口计算：查询侧用 ZUNIONSTORE 合并最近 60 个桶，
     可以灵活控制窗口大小（想改成 30 分钟只需改合并数量）。
  3. 自动过期：旧分钟的 ZSET 可以设 TTL（2小时），过了自动删除，
     不需要额外清理逻辑。
"""

from datetime import datetime, timezone


async def update_popularity(redis_client, video_id: int, delta: int):
    """
    更新视频热度分 — 写入对应分钟的滑动窗口 ZSET。

    调用时机：
      - 用户点赞 → delta=+1
      - 用户取消点赞 → delta=-1
      - 用户评论 → delta=+1
      - LikeWorker/CommentWorker 异步消费时 → delta=+1 或 -1

    参数：
      video_id: 视频 ID
      delta: 热度变化量（+1 或 -1）
    """
    if redis_client is None or video_id == 0 or delta == 0:
        return

    try:
        # 1. 失效视频详情缓存（数据变了，旧缓存不能用了）
        await redis_client.delete(f"v1:video:detail:id={video_id}")

        # 2. 获取当前 UTC 分钟字符串作为窗口 key
        now = datetime.now(timezone.utc)
        minute_key = f"v1:hot:video:1m:{now.strftime('%Y%m%d%H%M')}"

        # 3. ZINCRBY：增加该视频在当前分钟窗口的热度
        member = str(video_id)
        await redis_client.zincrby(minute_key, member, float(delta))

        # 4. 设置窗口 ZSET 的 TTL（2小时后自动删除）
        #    不依赖 SETEX，用 EXPIRE 单独设置（即使ZINCRBY已经创建了key）
        await redis_client.expire(minute_key, 7200)

    except Exception as e:
        logging.error(f"热度更新失败 video={video_id} delta={delta}: {e}")
        # 热度数据允许短暂不一致，失败不阻塞业务


async def get_hot_ranking(redis_client, limit: int = 50) -> list[int]:
    """
    获取当前热度排行榜（简化版，不包含快照分页）。
    合并最近 60 个分钟窗口，取 Top N。
    """
    if redis_client is None:
        return []

    now = datetime.now(timezone.utc).replace(second=0, microsecond=0)

    # 收集 60 个分钟窗口的 key
    window_keys = []
    for i in range(60):
        minute = now - timedelta(minutes=i)  # noqa: F821 (timedelta imported from datetime)
        window_keys.append(f"v1:hot:video:1m:{minute.strftime('%Y%m%d%H%M')}")

    # ZUNIONSTORE SUM 聚合
    dest_key = f"v1:hot:video:merge:1m:{now.strftime('%Y%m%d%H%M')}"
    await redis_client.zunionstore(dest_key, window_keys, aggregate="SUM")
    await redis_client.expire(dest_key, 120)

    # 取 Top N 的 video_id
    member_ids = await redis_client.zrevrange(dest_key, 0, limit - 1)
    return [int(m) for m in member_ids]
```

```python
# app/core/cache_service.py（分布式锁防击穿示例）

async def get_with_breakdown_protection(
    redis_client,
    cache_key: str,
    db_query_fn,      # 回源函数：async () -> data
    ttl: int = 300,
) -> Any:
    """
    带击穿防护的缓存读取。

    流程：
      1. 读缓存 → 命中返回
      2. 未命中 → SETNX 抢锁
          ├─ 抢到锁 → 查 DB → 写缓存 → 释放锁 → 返回
          └─ 没抢到 → 等待 N 次重试读缓存 → 超时则直接查 DB（降级）
    """
    # 1. 读缓存
    cached = await redis_client.get_json(cache_key)
    if cached is not None:
        return cached

    # 2. 缓存未命中 → 尝试获取分布式锁
    lock_key = f"lock:{cache_key}"
    lock_token = await redis_client.lock_acquire(lock_key, ttl=2)  # 锁 TTL 2秒

    if lock_token is not None:
        # ★ 抢到锁 — 我是回源者
        try:
            # 双重检查：抢到锁后再次读缓存（可能别人已经回源了）
            cached = await redis_client.get_json(cache_key)
            if cached is not None:
                return cached

            # 查 DB
            data = await db_query_fn()
            if data is not None:
                # 写缓存
                await redis_client.set_json(cache_key, data, ttl=ttl)
            return data
        finally:
            # ★ 释放锁（无论回源成功或失败都要释放）
            await redis_client.lock_release(lock_key, lock_token)
    else:
        # ★ 没抢到锁 — 等待别人回源
        for i in range(10):
            await asyncio.sleep(0.05)  # 等 50ms
            cached = await redis_client.get_json(cache_key)
            if cached is not None:
                return cached

        # 超时 → 降级为直接查 DB（不再等锁）
        logging.warning(f"缓存归源超时: {cache_key}，直接查DB")
        return await db_query_fn()
```

#### C.7 单飞（singleflight）的 Python 完整实现

```python
# app/core/singleflight.py
"""
Singleflight — 并发请求合并。

对应 Go 的 golang.org/x/sync/singleflight.Group

作用：同一个 key 的并发调用只执行一次，其余等待结果。

典型场景：
  1. ZSET 重建：100 个请求同时发现 ZSET 为空，只有 1 个去查 MySQL 重建
  2. 三级缓存 L3 回源：10 个请求同时 miss video_id=42，只查 1 次 MySQL
  3. 冷热边界缝合：多人同时翻到冷热边界，冷数据只查 1 次

线程安全说明：
  这个实现用 asyncio.Event 做同步，适用于单进程 asyncio 场景。
  singleflight 只保证同一进程内协程之间的去重。
  多进程（uvicorn --workers 4）时，每个进程最多各自查一次 MySQL。
"""

import asyncio
from typing import Callable, Awaitable, TypeVar

T = TypeVar("T")


class SingleFlight:
    def __init__(self):
        self._inflight: dict[str, asyncio.Event] = {}
        self._results: dict[str, any] = {}
        self._errors: dict[str, Exception | None] = {}

    async def do(self, key: str, fn: Callable[[], Awaitable[T]]) -> T:
        """
        执行 fn(key)。如果同一个 key 已经有正在执行的 fn，等待其结果。

        用法：
          result = await singleflight.do("rebuild_timeline", lambda: rebuild())

        工作流程：
          协程A: do("X", fn) → key "X" 不在 _inflight 中 → 创建 Event
                → 执行 fn → 结果存入 _results → Event.set()
          协程B: do("X", fn) → key "X" 在 _inflight 中 → Event.wait()
                → 被 A 唤醒 → 从 _results 拿结果
          协程C: do("X", fn) → key "X" 不在 _inflight 中了（A 已清理）
                → 重新执行 fn（说明 A 和 B 都已经拿到结果离开了）
        """
        # 如果当前 key 已经有正在执行的协程，等待它
        if key in self._inflight:
            event = self._inflight[key]
            await event.wait()  # ← 挂起，等执行者完成
            # 被唤醒后，从结果缓存中拿
            if key in self._errors and self._errors[key] is not None:
                raise self._errors[key]
            return self._results[key]

        # 我是第一个：创建 Event 标记"飞行中"
        self._inflight[key] = asyncio.Event()

        try:
            # 执行真正的函数
            result = await fn()
            self._results[key] = result
            self._errors[key] = None
            return result
        except Exception as e:
            self._errors[key] = e
            raise
        finally:
            # 唤醒所有等待者
            event = self._inflight.pop(key)
            event.set()

            # 延迟清理：给等待协程一点时间读取结果
            # 实际中可以用 asyncio 的 call_later
            await asyncio.sleep(0)
            self._results.pop(key, None)
            self._errors.pop(key, None)
```

### D. 面试追问预判

#### Q1: "缓存穿透你怎么解决？"

"两种方案，本项目中场景适用第一种：

1. **缓存空值**：查 DB 发现数据不存在时，缓存一个特殊的空标记（如 `{"_null": true}`），TTL 设短一些（1-5 分钟）。这样恶意请求同一个不存在的 ID 不会反复穿透到 DB。

2. **布隆过滤器**：用极小的内存存储所有存在的 video_id。请求先过布隆过滤器——如果布隆说'不存在'那一定不存在（不会穿透），如果布隆说'可能存在'才去查缓存→DB。适合 ID 空间大且有效 ID 相对固定的场景。

为什么本项目用方案 1？因为视频 ID 是自增的，前端不会大规模请求不存在的 ID（不像电商场景中恶意遍历优惠券码）。缓存空标记实现简单且足够应对。"

#### Q2: "Redis 内存不够了怎么办？"

"几个层次的应对：
1. **淘汰策略**：配置 `maxmemory-policy allkeys-lru`，Redis 内存满时自动淘汰最久未使用的 key
2. **压缩**：大 JSON 可以用 msgpack 替代 JSON，体积减少 30-50%
3. **限流**：L1 本地缓存挡掉 70% 的请求，减少对 Redis 的压力
4. **监控告警**：在内存使用到 80% 时触发告警，提前扩容或排查是否有泄漏
5. **ZSET 裁剪**：热榜窗口 ZSET 设 TTL 2 小时自动删除；`feed:global_timeline` ZSET 只保留 1000 条（ZREMRANGEBYRANK）。这两个是 Redis 内存的主要占用者。"

#### Q3: "为什么不用 Redis 做 Feed 流的全量缓存？"

"Feed 流数据的特点是**数据量大、变更频繁、个性化**——每个用户可能有不同的关注列表、不同的游标位置。全量缓存意味着缓存 Key 的组合爆炸（100 用户 × 1000 种分页组合 = 10 万个 Key），命中率很低。现在只缓存匿名流（所有人看到的都一样）和关注流的第一页（命中率最高），长尾数据让 MySQL 处理。"

#### Q4: "为什么 L1 本地缓存不用 Redis 替代？"

"Redis 再快也是网络 I/O（0.3-1ms 往返），L1 本地内存是进程内访问（纳秒级），快 1000 倍。L1 适合存热点中的热点——Feed 流中频繁出现的视频（如发了一小时的视频被 100 人刷到），每次命中 L1 就省一次 Redis 网络往返。但这只对单实例有效——多实例部署时 L1 数据不一致，所以 TTL 只设 3 秒。"

### A. 设计决策

#### A.1 RabbitMQ 在这个系统中的定位

**RabbitMQ 是可选的加速层，不是必须的依赖**。这句话是整个 MQ 设计的灵魂。

```
系统可用性降级链：
  最优：RabbitMQ + Redis 都可用 → 异步写入（最快）
  次优：RabbitMQ 挂、Redis 可用 → MySQL 直写 + Redis 热榜更新
  保底：RabbitMQ + Redis 都挂 → 纯 MySQL 运行（最慢但能用）
```

#### A.2 为什么选择 RabbitMQ 而不是 Kafka？

| 维度 | RabbitMQ | Kafka |
|------|----------|-------|
| 消息模型 | 推模式（Broker 推给 Consumer） | 拉模式（Consumer 主动拉） |
| 消息可靠性 | 每条消息独立 ACK，支持 DLX 死信 | 基于 offset 提交，消息不自动重试 |
| 部署复杂度 | 轻量（一个容器） | 重（ZK + Broker + 分区管理） |
| 适用量级 | 万级 msg/s | 百万级 msg/s |
| 路由灵活性 | Topic Exchange 灵活路由 | 只能按 topic 分区 |

**面试金句**：Kafka 是"分布式日志"（适合大数据流处理、事件溯源），RabbitMQ 是"消息代理"（适合任务分发、RPC 异步化）。我们这个项目的消息量级（每秒几十到几百条点赞/评论事件）RabbitMQ 绰绰有余，更重要的是它的死信队列和灵活路由开箱即用。

```python
# app/core/rabbitmq.py
"""
RabbitMQ 连接管理 — 基于 aio-pika 异步库。

对应 Go 的：
  internal/middleware/rabbitmq/rabbitMQ.go  — 连接 + PublishJSON
  internal/middleware/rabbitmq/likeMQ.go    — Like 事件生产者
  internal/middleware/rabbitmq/commentMQ.go — Comment 事件生产者
  internal/middleware/rabbitmq/socialMQ.go  — Social 事件生产者
  internal/middleware/rabbitmq/popularityMQ.go — Popularity 事件生产者
  internal/middleware/rabbitmq/timelineMQ.go   — Timeline 事件生产者
  internal/middleware/rabbitmq/dlx.go      — 死信队列
"""

import asyncio
import json
import logging
import secrets
from datetime import datetime, timezone
from typing import Optional
import aio_pika


# ============================================================
# 连接与拓扑声明
# ============================================================

class RabbitMQManager:
    """
    RabbitMQ 连接管理器。
    负责：建立连接 → 声明所有 Exchange/Queue/Binding → 提供 Producer 工厂
    """

    def __init__(self):
        self._connection: Optional[aio_pika.RobustConnection] = None
        self._channel: Optional[aio_pika.RobustChannel] = None

    async def connect(self, host: str, port: int, user: str, password: str):
        """
        建立 RabbitMQ 连接。
        aio_pika 的 RobustConnection 内置自动重连——
        网络断开后自动重新建立连接，对上层透明。
        """
        url = f"amqp://{user}:{password}@{host}:{port}/"
        self._connection = await aio_pika.connect_robust(url)
        self._channel = await self._connection.channel()
        # 设置 QoS：每次最多预取 50 条未确认消息（防止某个 Consumer 积压过多）
        await self._channel.set_qos(prefetch_count=50)
        logging.info("RabbitMQ connected")

    async def close(self):
        """关闭连接，释放 TCP socket"""
        if self._channel:
            await self._channel.close()
        if self._connection:
            await self._connection.close()

    @property
    def channel(self):
        return self._channel

    async def declare_topology(self, exchange_name: str, queue_name: str, binding_key: str):
        """
        声明一套 Exchange + Queue + Binding + DLX。
        这是所有业务 MQ 的通用模板。

        参数：
          exchange_name: 如 "like.events"
          queue_name:    如 "like.events"
          binding_key:   如 "like.*"  (topic 通配符)
        """
        # 1. 声明 Topic Exchange（持久化，服务重启不丢失）
        exchange = await self._channel.declare_exchange(
            exchange_name,
            aio_pika.ExchangeType.TOPIC,
            durable=True,      # 持久化：Broker 重启后 Exchange 还在
        )

        # 2. 声明死信 Exchange（全局共享一个 DLX）
        dlx = await self._channel.declare_exchange(
            "dlx.events",
            aio_pika.ExchangeType.TOPIC,
            durable=True,
        )

        # 3. 声明死信队列（每个业务队列对应一个 DLX 队列）
        dlx_queue = await self._channel.declare_queue(
            f"{queue_name}.dlx",
            durable=True,
        )
        # DLX 队列绑定所有消息（# 匹配任意 routing key）
        await dlx_queue.bind(dlx, routing_key="#")

        # 4. 声明业务队列，绑定 DLX
        queue = await self._channel.declare_queue(
            queue_name,
            durable=True,
            arguments={
                "x-dead-letter-exchange": "dlx.events",  # 死信转发到 DLX
            },
        )

        # 5. 绑定业务队列到业务 Exchange
        await queue.bind(exchange, routing_key=binding_key)

        logging.info(f"MQ topology ready: {exchange_name} → {queue_name} (binding: {binding_key})")
        return exchange, queue
```

#### C.2 事件生产者（Producer）

```python
# app/core/rabbitmq.py（续）

class EventProducer:
    """
    事件生产者基类 — 封装消息发布逻辑。
    所有业务 Producer（LikeProducer / CommentProducer / etc.）继承此类。
    """

    def __init__(self, channel, exchange_name: str):
        self._channel = channel
        self._exchange_name = exchange_name

    async def _publish(self, routing_key: str, payload: dict):
        """
        发布一条持久化消息到 RabbitMQ。

        delivery_mode=2 (PERSISTENT)：
          消息写入磁盘后再返回，保证 Broker 重启不丢消息。
          代价是吞吐量略低（磁盘 I/O），但对可靠性优先的场景是值得的。
        """
        message = aio_pika.Message(
            body=json.dumps(payload, ensure_ascii=False).encode("utf-8"),
            content_type="application/json",
            delivery_mode=aio_pika.DeliveryMode.PERSISTENT,  # 持久化
            timestamp=datetime.now(timezone.utc),
        )

        exchange = await self._channel.get_exchange(self._exchange_name)
        await exchange.publish(message, routing_key=routing_key)

    @staticmethod
    def _event_id() -> str:
        """生成唯一事件 ID（32字符 hex）"""
        return secrets.token_hex(16)


# ---- 各业务 Producer ----

class LikeProducer(EventProducer):
    """点赞事件生产者 — 对应 Go 的 LikeMQ"""

    def __init__(self, channel):
        super().__init__(channel, "like.events")

    async def publish_like(self, account_id: int, video_id: int):
        """发布点赞事件 → routing key: like.like"""
        await self._publish("like.like", {
            "event_id": self._event_id(),
            "action": "like",
            "user_id": account_id,
            "video_id": video_id,
            "occurred_at": datetime.now(timezone.utc).isoformat(),
```

```python
# app/workers/like_worker.py
"""
LikeWorker — 消费 like.events 队列，异步写 MySQL。

对应 Go 的 internal/worker/likeworker.go

消费逻辑：
  like 事件 → INSERT INTO likes (幂等，重复则忽略)
           → UPDATE videos SET likes_count = likes_count + 1
           → UPDATE videos SET popularity = popularity + 1
  unlike 事件 → DELETE FROM likes
             → UPDATE videos SET likes_count = GREATEST(likes_count - 1, 0)
             → UPDATE videos SET popularity = GREATEST(popularity - 1, 0)

重试策略：
  消费失败 → Nack (requeue=true) → RabbitMQ 重新投递
  失败 3 次后 → Ack (不 requeue) → 消息进入 DLX 死信队列
"""

import asyncio
import json
import logging
from datetime import datetime, timezone
import aio_pika
from sqlalchemy import select, update
from sqlalchemy.ext.asyncio import AsyncSession
from app.models.video import Video
from app.models.like import Like

MAX_RETRY = 3  # 最大重试次数


async def run_like_worker(channel: aio_pika.RobustChannel, session_factory):
    """
    LikeWorker 主循环。

    参数：
      channel: RabbitMQ channel（用于消费消息）
      session_factory: SQLAlchemy async session 工厂（每个消息一个独立事务）
    """
    # 声明拓扑并获取队列
    await channel.declare_exchange("like.events", aio_pika.ExchangeType.TOPIC, durable=True)
    queue = await channel.declare_queue(
        "like.events",
        durable=True,
        arguments={"x-dead-letter-exchange": "dlx.events"},
    )
    await queue.bind("like.events", routing_key="like.*")

    async with queue.iterator() as queue_iter:
        async for delivery in queue_iter:
            async with delivery.process(requeue_on_error=True):
                # ★ process() 上下文管理器：
                #   正常退出 → 自动 Ack
                #   抛异常  → 自动 Nack + requeue（回到队列尾部等待重试）
                await _handle_like_delivery(delivery, session_factory)


async def _handle_like_delivery(
    delivery: aio_pika.IncomingMessage,
    session_factory,
):
    """处理单条点赞消息"""
    body = delivery.body
    if not body:
        return  # 空消息直接跳过

    # 解析事件
    try:
        event = json.loads(body)
    except json.JSONDecodeError:
        logging.warning(f"LikeWorker: 消息解析失败，丢弃: {body[:100]}")
        return  # 解析失败直接丢弃（不重试，重试也没用）

    user_id = event.get("user_id", 0)
    video_id = event.get("video_id", 0)
    action = event.get("action", "")

    if not user_id or not video_id:
        return  # 无效消息直接丢弃

    # 检查重试次数（从 x-death header 中取）
    retry_count = _get_retry_count(delivery)
    if retry_count >= MAX_RETRY:
        # ★ 超过最大重试次数 → 抛异常让 process() 把消息 Nack
        #    因为 requeue=False（由 delivery.process 的参数控制），
        #    消息会被路由到 DLX 死信队列
        logging.error(
            f"LikeWorker: 超过最大重试次数 ({MAX_RETRY})，送入DLX: "
            f"action={action}, user={user_id}, video={video_id}"
        )
        raise RuntimeError(f"Max retry exceeded for like event: {action}")

    # 执行业务逻辑
    async with session_factory() as session:
        async with session.begin():
            if action == "like":
                await _apply_like(session, user_id, video_id)
            elif action == "unlike":
                await _apply_unlike(session, user_id, video_id)
            # 未知 action → 直接跳过（不重试）


def _get_retry_count(delivery: aio_pika.IncomingMessage) -> int:
    """
    从消息的 x-death header 中获取当前重试次数。
    RabbitMQ 在重新投递消息时会自动添加 x-death header，
    记录了该消息之前被消费失败的次数。
    """
    deaths = delivery.headers.get("x-death")
    if not deaths or not isinstance(deaths, list) or len(deaths) == 0:
        return 0
    first_death = deaths[0]
    if isinstance(first_death, dict):
        return int(first_death.get("count", 0))
    return 0


async def _apply_like(session: AsyncSession, user_id: int, video_id: int):
    """执行点赞：INSERT like + UPDATE 计数"""
    # 校验视频存在
    result = await session.execute(select(Video.id).where(Video.id == video_id))
    if result.scalar_one_or_none() is None:
        return  # 视频不存在 → 跳过（可能已被删除），不重试

    # INSERT like，重复则忽略（联合唯一索引保证幂等）
    from sqlalchemy.dialects.mysql import insert
    stmt = insert(Like).values(
        video_id=video_id,
        account_id=user_id,
        created_at=datetime.now(timezone.utc),
    )
    # ON DUPLICATE KEY UPDATE：如果已存在（重复点赞），不报错，静默忽略
    stmt = stmt.on_duplicate_key_update(id=Like.id)

    result = await session.execute(stmt)
    if result.rowcount == 0:
        return  # 重复点赞（联合唯一索引命中），静默跳过

    # likes_count + 1
    await session.execute(
        update(Video)
        .where(Video.id == video_id)
        .values(likes_count=Video.likes_count + 1)
    )
    # popularity + 1
    await session.execute(
        update(Video)
        .where(Video.id == video_id)
        .values(popularity=Video.popularity + 1)
    )

    logging.debug(f"LikeWorker: user={user_id} liked video={video_id}")


async def _apply_unlike(session: AsyncSession, user_id: int, video_id: int):
    """执行取消点赞：DELETE like + UPDATE 计数"""
    # 删除点赞记录
    result = await session.execute(
        select(Like).where(Like.video_id == video_id, Like.account_id == user_id)
    )
    like = result.scalar_one_or_none()
    if like is None:
        return  # 记录不存在 → 可能已经取消过了，静默跳过

    await session.delete(like)

    # likes_count - 1（GREATEST 防负数）
    await session.execute(
        update(Video)
        .where(Video.id == video_id)
        .values(likes_count=Video.likes_count - 1)  # 实际 SQL 中会加 GREATEST
    )
    # popularity - 1
    await session.execute(
        update(Video)
        .where(Video.id == video_id)
        .values(popularity=Video.popularity - 1)
    )

    logging.debug(f"LikeWorker: user={user_id} unliked video={video_id}")
```

```python
        "video.timeline.update.queue",
        durable=True,
        arguments={"x-dead-letter-exchange": "dlx.events"},

    async with queue.iterator() as queue_iter:
        async for delivery in queue_iter:
            async with delivery.process():
                try:
                    event = json.loads(delivery.body)
```

```python
# app/workers/main.py
"""
Worker 进程入口 — 启动所有 MQ 消费者。
对应 Go 的 cmd/worker/main.go

这个文件作为独立进程运行（不是 FastAPI 的一部分）：
  python -m app.workers.main

启动流程：
  1. 连接 MySQL（带重试）
  2. 连接 Redis（可选）
  3. 连接 RabbitMQ（带重试）
  4. 声明所有 MQ 拓扑
  5. 并发启动所有 Worker 协程
  6. 监听 SIGTERM 优雅关闭
"""

import asyncio
import logging
import signal

from app.config import settings
from app.database import create_engine, close_engine, SessionLocal
from app.core.redis import create_redis, close_redis
from app.core.rabbitmq import RabbitMQManager
from app.core.rabbitmq import (
    LikeProducer, CommentProducer, SocialProducer,
    PopularityProducer, TimelineProducer,
)
from app.workers.like_worker import run_like_worker
from app.workers.comment_worker import run_comment_worker
from app.workers.social_worker import run_social_worker
from app.workers.popularity_worker import run_popularity_worker
from app.workers.timeline_consumer import run_timeline_consumer
from app.workers.outbox_worker import run_outbox_poller


async def connect_with_retry(name: str, max_retries: int, factory):
    """指数退避重连：1s → 2s → 4s → ... 最多 30s"""
    for i in range(max_retries):
        try:
            return await factory()
        except Exception as e:
            wait = min(2 ** i, 30)
            logging.warning(f"{name} 不可用，{wait}s 后重试 ({i+1}/{max_retries}): {e}")
            await asyncio.sleep(wait)
    raise RuntimeError(f"{name}: 超过最大重试次数")


async def main():
    logging.basicConfig(level=logging.INFO)

    # 1. 连接 MySQL
    await connect_with_retry("MySQL", 10, create_engine)

    # 2. 连接 Redis（可选）
    redis_client = None
    try:
        redis_client = await connect_with_retry("Redis", 3, create_redis)
        logging.info("Redis connected (Worker)")
    except Exception:
        logging.warning("Redis not available (Worker continues without it)")

    # 3. 连接 RabbitMQ
    rmq = RabbitMQManager()
    await connect_with_retry("RabbitMQ", 10, lambda: rmq.connect(
        settings.rabbitmq.host, settings.rabbitmq.port,
        settings.rabbitmq.username, settings.rabbitmq.password,
    ))

    # 4. 声明所有 MQ 拓扑
    channel = rmq.channel
    await rmq.declare_topology("like.events", "like.events", "like.*")
    await rmq.declare_topology("comment.events", "comment.events", "comment.*")
    await rmq.declare_topology("social.events", "social.events", "social.*")
    await rmq.declare_topology("video.popularity.events", "video.popularity.events", "video.popularity.*")
    await rmq.declare_topology("video.timeline.events", "video.timeline.update.queue", "video.timeline.*")

    # 5. 并发启动所有 Worker
    tasks = [
        asyncio.create_task(run_like_worker(channel, SessionLocal), name="like_worker"),
        asyncio.create_task(run_comment_worker(channel, SessionLocal), name="comment_worker"),
        asyncio.create_task(run_social_worker(channel, SessionLocal), name="social_worker"),
        asyncio.create_task(run_popularity_worker(channel, redis_client), name="popularity_worker"),
        asyncio.create_task(run_timeline_consumer(channel, redis_client), name="timeline_consumer"),
        # Outbox Poller 只在 Worker 进程中启动（API Server 不启动，避免重复轮询）
        asyncio.create_task(run_outbox_poller(), name="outbox_poller"),
    ]

    logging.info(f"All workers started ({len(tasks)} consumers)")

    # 6. 优雅关闭：捕获 SIGTERM/SIGINT → 取消所有协程 → 关闭连接
    loop = asyncio.get_running_loop()
    stop_event = asyncio.Event()

    def _signal_handler():
        logging.info("Received shutdown signal, stopping workers...")
        stop_event.set()

    for sig in (signal.SIGTERM, signal.SIGINT):
        try:
            loop.add_signal_handler(sig, _signal_handler)
        except NotImplementedError:
            # Windows 不支持 add_signal_handler，用 signal.signal 替代
            signal.signal(sig, lambda *args: stop_event.set())

    await stop_event.wait()

    # 取消所有运行中的 Worker 协程
    for task in tasks:
        task.cancel()
    await asyncio.gather(*tasks, return_exceptions=True)

    # 关闭连接
    await rmq.close()
    await close_redis(redis_client)
    await close_engine()
    logging.info("Worker shutdown complete")


if __name__ == "__main__":
    asyncio.run(main())
```

> **OS 知识点 — 优雅关闭**：当 Docker 执行 `docker stop` 时，容器内的主进程（PID 1）先收到 SIGTERM。如果 10 秒内没退出，Docker 会发 SIGKILL 强制杀进程。优雅关闭的关键是：收到 SIGTERM → 停止接受新消息 → 处理完当前正在消费的消息 → 关闭 TCP 连接 → 退出。上面的 `stop_event.wait()` + `task.cancel()` + `asyncio.gather` 实现了这个流程。

### D. 面试追问预判

#### Q1: "消息丢了怎么办？RabbitMQ 怎么保证消息不丢？"

"三个层面防止消息丢失：

1. **Producer 端**：消息设置 `delivery_mode=PERSISTENT`，RabbitMQ 写入磁盘后才返回确认。如果 MQ 发布失败（网络异常、Broker 宕机），Producer 捕获异常后降级为直接写 MySQL（第 4 层的异步优先+降级兜底模式）。

2. **Broker 端**：Exchange 和 Queue 都声明为 `durable=True`，Broker 重启后拓扑和消息都不丢。

3. **Consumer 端**：手动 Ack（`auto_ack=False`），只有业务处理成功后才 Ack。如果处理过程中进程崩溃，消息因为没有 Ack 会被 RabbitMQ 重新投递给其他 Consumer。

4. **最后防线 — 死信队列**：消费失败 3 次后消息进入 DLX，不会无限重试也不会静默丢弃。运维可以从 DLX 中检查失败消息并手动处理。"

#### Q2: "RabbitMQ 和 Kafka 的 ACK 机制有什么区别？"

"RabbitMQ 是**消息级 ACK**——每条消息独立确认，Consumer 处理一条确认一条。Kafka 是 **offset 提交**——Consumer 定期提交'我处理到了 offset N'，如果提交后 Crash，offset 之后的所有消息都不会重新消费（at-most-once），但如果处理完 N+1 条但还没提交 offset 就 Crash，重启后会重复消费这批消息（at-least-once）。

RabbitMQ 的模式更适合我们这个场景——点赞/评论每条消息独立，失败的消息单独重试，不影响其他消息。Kafka 更适合需要批处理和高吞吐的场景。"

#### Q3: "消费失败的消息一直在重试，会不会阻塞后面的消息？"

"不会。RabbitMQ 的 Nack + requeue 是把消息放回队列尾部，后面的消息可以继续被消费。但如果某条消息反复失败（比如外键约束冲突），它会在'消费→Nack→回到队尾→再次被消费'中循环。这就是为什么需要**最大重试次数**——3 次后不再 requeue，直接 Ack 把消息移除（由 DLX 接管）。这样坏消息不会无限阻塞队列。"

#### Q4: "为什么 Worker 不放在 API 进程里，而要单独部署？"

"三个原因：
1. **资源隔离**：Worker 批量写 MySQL 时消耗数据库连接和 CPU，和 API Server 共享一个进程会互相影响。API 需要快速响应，Worker 可以慢慢处理。
2. **独立扩缩容**：高峰期 Worker 积压时可以单独加实例，API Server 不需要跟着扩。
3. **故障隔离**：Worker 挂了不影响用户请求（降级为直写 MySQL），API Server 挂了 Worker 把剩余消息消费完就停，不会产生脏数据。

从操作系统角度看，这是**进程级隔离**——两个进程有独立的内存空间、独立的文件描述符表、独立的信号处理。一个进程 Segmentation Fault 不会影响另一个。"
